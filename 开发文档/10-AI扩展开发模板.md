# 10-AI扩展开发模板

## 0. 需求来源与开发动因

- 业务价值摘要：提供可复用实施模板，降低AI功能迭代的人力成本。
- 业务背景：AI 能力迭代频繁，团队需要可复制模板承接新增测试类型和生成策略。
- 现状痛点：每次扩展都从零梳理，易导致实现风格不一致、测试遗漏和上线风险上升。
- 建设目标：沉淀“需求到交付”的标准实施模板与固定步骤。
- 预期收益：提高扩展效率与质量一致性，降低新增能力接入成本。

## 1. 模板用途

本文档是 AI 能力扩展的“执行模板”，用于团队在新增 AI 测试类型或增强生成能力时，按统一步骤开发、测试与交付，降低改动遗漏与回归风险。

---

## 2. 适用场景

- 新增一种 `test_type`（如“兼容性测试”“可用性测试”）
- 扩展已有类型的上下文输入（`ext_config` / `context_data`）
- 调整 Prompt 策略、RAG 拼接规则或去重策略
- 增加新的 AI 输出字段并映射到业务模型

---

## 3. 标准实施流程（Checklist）

### 3.1 需求定义

- [ ] 明确目标测试类型与输出对象（`TestCase` 或 `ApiTestCase`）
- [ ] 明确输入字段（前端传参）与输出字段（后端归一化）
- [ ] 明确是否启用 RAG、去重、流式输出
- [ ] 明确验收标准（功能、性能、可观测性）

### 3.2 后端代码改造

- [ ] 在 `assistant/views.py` 注册类型：
  - `_AI_TEST_TYPE_ALLOWED`
  - `_AI_TEST_TYPE_FOCUS`
- [ ] 在 `_prepare_ai_generate_context()` 增加新类型上下文合并逻辑（如需）
- [ ] 在 `assistant/services/ai_engine.py` 增加领域策略模板
- [ ] 在 `assistant/ai_generate_strategies.py` 补充分发与用户消息拼装
- [ ] 如需结构化增强：在 `assistant/api_case_generation.py` 补充清洗/映射
- [ ] 检查 `assistant/rag_pipeline.py` 是否需要新增去重规则补充
- [ ] 评估 `assistant/generated_case_dedup.py` 阈值是否需要按类型调整

### 3.3 前端代码改造

- [ ] 新增类型选项与 UI 控件
- [ ] 将扩展参数写入 `ext_config` 或 `context_data`
- [ ] 确认同步/流式调用均可覆盖新增类型
- [ ] 对新字段做兼容展示（避免空值报错）

### 3.4 测试与验证

- [ ] 补充 `assistant/tests.py` 单元测试
- [ ] 增加最小集成验证（`/api/ai/generate-cases/`）
- [ ] 验证流式接口事件收敛（`done` 可达）
- [ ] 验证无 key / 非法 spec / 异常参数分支
- [ ] 验证去重前后数量与内容合理性

### 3.5 文档与交付

- [ ] 更新 `开发文档/09-AI开发指南.md`
- [ ] 如接口约定变化，更新 `开发文档/04-API接口与数据交互.md`
- [ ] 如部署配置变化，更新 `开发文档/06-部署与配置指南.md`
- [ ] 记录改动范围、已知限制与回滚方案

## 4. 文件级改造模板

### 4.1 `assistant/views.py`

```python
# 1) 注册允许类型
_AI_TEST_TYPE_ALLOWED = frozenset({
    "functional", "api", "performance", "security", "ui-automation",
    "compatibility",  # 新增类型示例
})

# 2) 新增类型说明
_AI_TEST_TYPE_FOCUS = {
    # ...
    "compatibility": "兼容性测试：浏览器/系统/分辨率组合与降级策略（简体中文）。",
}
```

### 4.2 `assistant/services/ai_engine.py`

```python
PROMPT_TEMPLATES = {
    # ...
    "compatibility": (
        "【兼容性测试策略】按环境矩阵输出可执行用例，覆盖主流浏览器/系统/分辨率，"
        "明确兼容预期与降级行为。"
    ),
}
```

### 4.3 `assistant/ai_generate_strategies.py`

```python
def apply_test_type_domain_strategy(base_system_prompt: str, ctx: dict) -> str:
    tt = str(ctx.get("test_type") or "").strip()
    if tt == "compatibility":
        return base_system_prompt + "\n\n【兼容性补充】请输出环境矩阵与回归优先级。"
    return base_system_prompt
```

## 5. 测试用例骨架（可直接复制）

### 5.1 单元测试骨架（`assistant/tests.py`）

```python
def test_dispatch_ai_generation_for_compatibility():
    ctx = {
        "test_type": "compatibility",
        "module_name": "支付中心",
        "requirement": "覆盖 Chrome/Edge/Safari 在支付流程中的兼容性",
        "test_type_focus": "",
    }
    d = dispatch_ai_generation(ctx)
    assert d.system_prompt
    assert d.user_message
    assert d.should_enrich_api is False
```

### 5.2 接口验证骨架（pytest）

```python
def test_ai_generate_cases_compatibility(authed_client):
    payload = {
        "testType": "compatibility",
        "requirement": "覆盖移动端和桌面端支付页兼容性",
    }
    resp = authed_client.post(f"{authed_client.base_url}/ai/generate-cases/", json=payload, timeout=60)
    assert resp.status_code in (200, 400)
```

## 6. 提交前自检清单

- [ ] 代码中所有新增 `test_type` 字符串拼写一致
- [ ] 缺省分支不会抛异常（空 requirement、空 ext_config）
- [ ] 流式接口未破坏旧事件协议（`connected/meta/delta/ping/done`）
- [ ] 模型返回异常时可给出用户可读错误信息
- [ ] 文档、路由、前端调用路径已同步
- [ ] 本地至少完成一次 smoke 验证

---

## 7. 回滚与兼容建议

1. 新类型上线初期建议“灰度入口 + 可配置开关”
2. 保留默认类型回退（非法值回落 `functional`）
3. 结构化字段新增应保持向后兼容（旧字段不删除）
4. 去重阈值调整需记录版本，以便故障时快速恢复

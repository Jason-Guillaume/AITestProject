# 09-AI开发指南

## 0. 需求来源与开发动因

- 业务价值摘要：规范AI能力研发路径，保障扩展速度与质量稳定。
- 业务背景：AI 能力已成为测试平台核心增量，需要统一开发方法支撑持续扩展。
- 现状痛点：Prompt、模型调用、流式输出与 RAG 逻辑分散实现时，质量稳定性和回归风险难控制。
- 建设目标：将 AI 开发流程模板化、规范化，明确设计、实现、测试与发布路径。
- 预期收益：新能力接入更快、质量更稳、后续维护成本更低。

## 1. 文档目标

本文档面向本项目 AI 模块的开发者，聚焦“如何在现有架构上安全、稳定、可扩展地开发 AI 能力”。覆盖内容包括：

- AI 路由与核心调用链
- 请求上下文解析与参数规范
- Prompt 策略分发与领域扩展
- RAG 检索与去重机制
- 同步/流式生成接口实现要点
- 配置管理、测试策略与排错方法

## 3. 生成调用链路

### 3.1 同步生成链路

```
前端 POST /api/ai/generate-cases/
  ↓
_prepare_ai_generate_context() 解析请求上下文
  ↓
dispatch_ai_generation() 按 test_type 选择策略
  ↓
build_rag_system_prompt() 拼接检索结果与去重规则
  ↓
OpenAI Chat Completions 调用
  ↓
_parse_ai_cases_json() 解析模型返回
  ↓
_normalize_generated_case() 字段归一化
  ↓
（API类型）enrich_normalized_case_with_api_fields() 增强
  ↓
deduplicate_generated_cases() 去重
  ↓
semantic_deduplicate_cases() 与库内用例语义比对去重
  ↓
返回 { success, cases, model, message }
```

### 3.2 流式生成链路

```
前端 fetch /api/ai/generate-cases-stream/
  ↓
Request + JSONParser + TokenAuthentication 手动鉴权
  ↓
准备上下文并发起模型流式请求
  ↓
服务端分块输出 SSE data: {type, ...}
  ↓
前端按事件类型消费（connected/meta/delta/ping/done）
```

## 4. 请求参数与上下文规范

### 4.1 核心输入字段

`_prepare_ai_generate_context()` 支持多别名解析：

| 语义 | 可用字段 |
|------|------|
| 需求描述 | `prompt_text` / `requirement` / `requirement_description` / `description` |
| 接口定义 | `api_spec` / `api_interface_doc` / `openapi_spec` |
| 测试类型 | `testType` / `test_type` |
| 模块 | `module_id` / `module` / `module_name` |
| AI Key | `api_key` |
| 模型地址 | `api_base_url` / `base_url` |
| 模型名 | `model` |
| 扩展上下文 | `context_data` / `ext_config` |

### 4.2 test_type 允许值

- `functional`
- `api`
- `performance`
- `security`
- `ui-automation`

若传入非法值，默认回退到 `functional`。

### 4.3 上下文补全机制

`_prepare_ai_generate_context()` 会将 `ext_config` 映射为兼容 `context_data` 的旧字段，再拼入需求文本，确保 Prompt 与 RAG 都能消费补充信息（如性能指标、漏洞类型、UI 定位参考等）。

## 5. Prompt 策略与领域扩展

### 5.1 策略分发入口

统一通过 `dispatch_ai_generation(ctx)` 返回：

- `system_prompt`
- `user_message`
- `should_enrich_api`
- `prompt_context`

### 5.2 API 类型策略

当 `test_type == "api"` 时：

1. 使用 `api_case_generation.py` 的 API 专用 System Template
2. 强制输出 JSON 数组（当前业务约束为 1 条）
3. 后处理阶段映射到 `ApiTestCase` 结构化字段：
   - `api_method`
   - `api_url`
   - `api_headers`
   - `api_body`
   - `api_expected_status`

### 5.3 其他类型策略

`services/ai_engine.py` 的 `PROMPT_TEMPLATES` 提供领域策略补充：

- 功能测试：业务流程、验收规则
- 性能测试：并发、吞吐、时延与 SLA
- 安全测试：OWASP/攻击向量/防御预期
- UI 自动化：定位策略、等待与断言可脚本化

## 6. RAG 与去重机制

### 6.1 RAG 检索范围

`rag_chroma.py` 使用 Chroma 持久化集合，仅在**同一模块（module_id）**内检索 Top-K 相似用例，避免跨模块噪声。

向量文本来源（`build_testcase_embed_text`）：

- 用例名称
- 等级
- API 方法与 URL（API 类型）
- 步骤与预期

### 6.2 RAG 提示拼装

`rag_pipeline.py` 在 System Prompt 追加：

1. 相似用例块（Top 5）
2. 强制去重规则
3. `[ALL_COVERED]` 约定（若已完全覆盖）

同步接口在检测到 `[ALL_COVERED]` 时返回：

- `success: true`
- `all_covered: true`
- `cases: []`

### 6.3 生成结果去重

`generated_case_dedup.py` 通过两类相似度对比：

- 归一化 Levenshtein
- 词级 Jaccard

按阈值（默认 `0.8`）保留首条，移除后续近重复项。

### 6.4 语义去重（新增）

`services/semantic_dedup.py` 提供“生成后语义去重”能力：

1. 将“标题 + 步骤”拼接为嵌入文本
2. 调用 Embedding 接口向量化（当前复用 `assistant/embeddings_zhipu.py`）
3. 使用余弦相似度计算新用例与库内用例相似度
4. 若最高相似度 `>= 0.85`（可配置）则判重剔除

该机制与字符串去重互补：

- 字符串去重：处理字面重复
- 语义去重：处理措辞不同但意图重复

---

## 6.5 批量生成与 JSON Mode（新增）

新增 `services/case_batch_generation.py`，核心策略如下：

1. Prompt 强制模型一次输出不少于 5 条用例（可配置）
2. 必须覆盖正向、逆向、边界场景
3. 强制输出合法 JSON Array（仅输出 JSON 本体）
4. 优先开启 JSON Mode（`response_format={"type":"json_object"}`）
5. 若上游兼容端不支持 JSON Mode，自动回退普通调用并做容错解析

### 6.6 业务编排链路（新增）

同步与流式接口均已对齐以下编排：

1. 生成前规避：读取当前模块已有用例标题注入 Prompt
2. 批量生成：一次生成多条并解析成对象列表
3. 生成后校验：字符串去重 + 语义去重双层过滤
4. 返回前端：输出最终清洗后的多条用例

## 7. 连接与模型配置机制

### 7.1 配置优先级

1. 请求体显式传入 `api_key`（优先）
2. 否则读取系统配置 `AIModelConfig`（且 `is_connected=True`）

### 7.2 系统级配置接口

由 `user/sys_views.py` 提供：

- `GET /api/sys/ai-config/`
- `PUT /api/sys/ai-config/`
- `POST /api/sys/ai-config/disconnect/`
- `POST /api/sys/ai-config/reconnect/`
- `DELETE /api/sys/ai-config/`

写操作要求系统管理员权限。

### 7.3 特殊模型路由（讯飞 MaaS）

`assistant/views.py` 对 `iflytek-spark-maas-coding` 做了模型名与 base_url 归一化处理：

- 业务模型类型 -> 实际 payload 模型
- SDK 基础地址归一化

该逻辑已被 `assistant/tests.py` 覆盖。

## 8. 流式 SSE 开发要点

### 8.1 为什么使用 Django `View`

`AiGenerateCasesStreamView` 使用标准 Django `View`（非 DRF APIView），避免 `Accept: text/event-stream` 触发 DRF 内容协商导致 `406`。

### 8.2 鉴权方式

流式接口内手动调用：

- `Request(request, parsers=[JSONParser()])`
- `TokenAuthentication().authenticate(drf_request)`

保持与普通 DRF Token 一致。

### 8.3 事件协议建议

当前事件类型包含：

- `connected`
- `meta`
- `delta`
- `ping`
- `done`

新增事件类型时应保持向后兼容，并在前端 `assistant.js` 的流解析逻辑中同步处理。

## 9. 扩展开发指南

### 9.1 新增一种 test_type

建议步骤：

1. 在 `views.py` 的 `_AI_TEST_TYPE_ALLOWED` 与 `_AI_TEST_TYPE_FOCUS` 注册类型
2. 在 `ai_engine.py` 增加策略模板与扩展字段映射
3. 在 `ai_generate_strategies.py` 中补充分发与上下文拼装
4. 在前端补充类型选项与 `ext_config` 入参
5. 在 `assistant/tests.py` 增加对应分支测试

### 9.2 扩展 API 用例字段

优先修改 `api_case_generation.py`：

- 清洗函数：`sanitize_ai_raw_item_for_apitestcase`
- 映射函数：`enrich_normalized_case_with_api_fields`

保证“模型输出字段漂移”时后端仍可兼容旧格式。

### 9.3 调整去重策略

在 `generated_case_dedup.py` 调整：

- `DEFAULT_SIMILARITY_THRESHOLD`
- 相似度算法组合

建议同步补充基准测试，避免阈值过低导致误删。

## 10. 测试与质量保障

### 10.1 现有测试重点

`assistant/tests.py` 已覆盖：

- 上下文解析与 Spec 校验
- Prompt 路由与用户消息构造
- API 字段归一化与增强
- 模型连接路由特殊分支（含 Mock）
- 批量生成 Prompt/JSON 解析
- 语义去重核心逻辑（Mock Embedding）

### 10.2 新能力测试建议

1. 优先纯函数测试（策略、解析、映射）
2. 外部调用统一 Mock（避免 CI 抖动）
3. 对流式接口至少覆盖：
   - 鉴权失败
   - JSON 解析失败
   - `done` 事件收敛

## 11. 常见问题排查

| 问题 | 现象 | 排查建议 |
|------|------|------|
| 缺少 API Key | 生成接口直接报错 | 检查请求 `api_key` 或系统 `AIModelConfig` 是否已连接 |
| Spec 无法识别 | API 类型生成被拒绝 | 检查 `api_spec` 是否为合法 JSON/YAML/cURL |
| 返回无法解析 JSON | 同步生成失败 | 检查模型输出是否包含代码围栏或数组外文本 |
| 流式断流 | 前端中途停止 | 检查代理超时、SSE 事件解析、上游模型连接 |
| 结果重复 | 生成多条相似用例 | 调整 dedup 阈值或增强 Prompt 去重约束 |

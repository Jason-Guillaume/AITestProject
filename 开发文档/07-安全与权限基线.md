# 07-安全与权限基线

## 0. 需求来源与开发动因

- 业务价值摘要：前置安全规则与权限边界，降低数据与合规风险。
- 业务背景：平台涉及用户组织、测试资产、环境变量与 AI 配置等敏感数据，安全要求持续提升。
- 现状痛点：缺少权限和数据边界约束时，容易出现越权访问、敏感泄露和审计困难。
- 建设目标：前置认证授权、数据隔离、密钥治理与审计要求，建立统一安全底线。
- 预期收益：在系统扩展期保持安全可控与合规可审计。

## 1. 安全基线目标

本章定义项目在开发、测试、生产三个阶段的统一安全底线，确保系统在认证、授权、数据隔离、配置管理与运维环节具备可审计、可执行的安全标准。

---

## 2. 认证与会话基线

### 2.1 认证机制

后端基于 DRF Token 认证：

- 默认认证类：`TokenAuthentication`
- 默认权限类：`IsAuthenticated`

所有受保护接口必须携带：

```http
Authorization: Token <token>
```

开放接口仅限必要场景（如登录、注册、验证码）。

### 2.2 客户端令牌策略

前端通过 `request.js` 注入 Token 并处理失效：

1. 统一从 `localStorage.token` 读取并注入请求头
2. 响应 `401` 时立即清理 token 并跳转登录页
3. 禁止在 URL 或日志中打印 token 明文

## 3. 授权与权限边界

### 3.1 权限模型

系统采用“全局登录 + 角色权限”组合：

- `IsSystemAdmin`：系统管理员权限（组织、角色、系统用户等管理）
- `IsApprovalAdmin`：审批管理员权限（`is_superuser` / `is_staff` / `is_system_admin`）

### 3.2 前端路由权限

路由守卫限制系统管理页仅管理员可访问：

- `/system/org`
- `/system/role`
- `/system/user`
- `/system/messages`

非管理员访问时自动重定向到 `/system/message`。

### 3.3 最小权限原则

新增接口时应遵循：

1. 默认继承 `IsAuthenticated`
2. 涉及系统配置、审批、组织结构的接口必须显式增加管理员权限
3. 仅在业务必要时开放匿名访问

## 4. 数据隔离与访问控制

`BaseModelViewSet` 已实现普通用户数据范围收敛（Data Scope）：

- 优先按项目成员关系放行（`project/module/testcase/release/plan` 等链路）
- 兜底仅可见本人创建数据
- 管理员可见全量未删除数据

该机制是防止“跨项目数据越权读取”的核心保护层，新增模型时应优先复用此基类。

### 4.1 关系字段判定（重要修复点）

在实际迭代中出现过“字段同名但类型不同”导致的 500（例如某模型存在 `version` 字段但为 `CharField`，而隔离逻辑误按外键链路拼接 `version__project__...`）。

因此基线约束补充为：

- 数据隔离链路（如 `project__members`、`module__project__members` 等）**仅可对关系字段（FK/O2O/M2M）使用**。
- 当字段名存在但不是关系字段时，必须回退到 `creator=user` 的兜底策略，避免 `FieldError`。

对应实现已在 `common/views.py` 中增加“仅关系字段才 join”的保护。

## 5. 配置与密钥安全

### 5.1 配置项安全要求

生产环境必须满足：

- `DEBUG=False`
- `ALLOWED_HOSTS` 明确配置
- `SECRET_KEY` 不可写入仓库明文
- 数据库密码通过环境变量注入
- Elasticsearch/Loki 等外部依赖的账号密码通过环境变量注入，禁止在代码中提供可用默认值

#### 5.1.1 生产安全启动校验（新增）

为降低“配置漂移导致的隐性安全风险”，后端在 `DJANGO_DEBUG=0`（生产模式）下执行启动强校验：

- 若 `DJANGO_SECRET_KEY` 缺失 → **启动失败**
- 若 `DB_PASSWORD` 缺失 → **启动失败**
- 若设置了 `ELASTICSEARCH_USER` 但 `ELASTICSEARCH_PASSWORD` 缺失 → **启动失败**

该策略的目标是“宁可启动失败，也不带着不安全默认值运行”。

### 5.2 关键敏感信息

以下信息禁止硬编码或写入公开文档：

- `SECRET_KEY`
- 数据库账号密码
- 第三方 AI 平台 API Key
- 邮件 SMTP 密码

### 5.3 AI 配置管理

系统级 AI 配置通过 `/api/sys/ai-config/` 管理，权限为系统管理员。

要求：

1. API Key 仅在受控环境录入
2. 关闭不再使用的模型连接
3. 变更记录纳入运维审计

---

## 6. 输入校验与序列化保护

当前系统通过 Serializer 执行字段校验与格式约束，例如：

- 密码、用户名变更申请校验
- 性能任务并发与持续时间格式校验
- 上传文件类型与大小限制

新增接口时应确保：

1. 参数校验在 Serializer 层完成
2. 非法输入返回明确 4xx 错误
3. 不将底层异常栈直接暴露给前端

---

## 7. 缓存、验证码与降级策略

验证码依赖缓存服务，系统支持 Redis 不可用时降级 LocMem：

- 优点：保障服务连续性
- 风险：多实例下验证码状态不共享

生产建议：

1. 强制启用 Redis 集中缓存
2. 监控 Redis 可用性与连接池
3. 验证码过期时间与频率限制纳入风控策略

---

## 8. 部署安全基线

### 8.1 传输安全

- 全站启用 HTTPS
- 反向代理层禁用不安全协议与弱加密套件

### 8.2 网络与访问面

- 数据库与 Redis 仅内网可访问
- 管理接口仅对可信网络开放
- 限制管理后台与 SSH 入口来源 IP

### 8.3 日志与审计

- 登录失败、权限拒绝、审批动作、配置变更需可追踪
- 日志避免输出敏感字段明文（token、密码、key）

# 12-定时调度执行模块开发文档

## 0. 需求来源与开发动因

- 业务价值摘要：实现回归自动化与可审计执行，提升测试运营效率。
- 业务背景：团队希望将重复性回归测试从人工触发升级为定时自动执行。
- 现状痛点：手工执行易遗漏且不可追踪，难以形成稳定的周期质量监控机制。
- 建设目标：基于 Cron 调度与执行日志沉淀，建立自动化回归链路。
- 预期收益：提升执行效率与稳定性，实现可持续、可审计的回归闭环。

## 1. 模块目标

定时调度执行模块用于将测试任务按 Cron 规则自动触发执行，并记录执行历史，形成可追踪、可审计的自动化执行闭环。

核心能力：

- 调度任务 CRUD 与状态控制
- Cron 规则解析与触发
- 动态增删改暂停恢复任务
- 任务执行日志追踪

## 2. 技术方案

### 2.1 调度组件

- `APScheduler`：任务调度引擎
- `django-apscheduler`：与 Django 深度集成（任务持久化、管理命令）
- `CronTrigger.from_crontab()`：支持标准 5 段 Cron 表达式

### 2.2 集成位置

- 新增文件：`execution/scheduler.py`
- 通过 `ExecutionConfig.ready()` 启动调度器（带防重复保护）

## 3. 数据模型设计

### 3.1 ScheduledTask

建议表名：`scheduled_task`

关键字段：

- `name`: 任务名称
- `cron_expression`: Cron 表达式（如 `0 2 * * *`）
- `status`: 状态（active/paused/disabled）
- `job_id`: APScheduler 任务唯一 ID
- `environment`: 关联测试环境
- `test_cases`: 关联测试用例（M2M）
- `last_run_time`: 最近执行时间
- `next_run_time`: 下次计划执行时间
- `last_status`: 最近执行结果（success/failed/partial）
- `last_message`: 最近执行摘要
- 继承 `BaseModel`

### 3.2 ScheduledTaskLog

建议表名：`scheduled_task_log`

关键字段：

- `scheduled_task`: 关联调度任务
- `trigger_time`: 触发时间
- `start_time/end_time`: 执行起止
- `status`: 执行状态（success/failed/running/partial）
- `message`: 执行摘要
- `detail`: 明细（JSON）
- 继承 `BaseModel`

## 4. 调度逻辑设计

## 4.1 TestScheduler 对外方法

- `start()`：启动调度器
- `shutdown()`：关闭调度器
- `sync_task(task)`：根据 DB 任务状态刷新 APScheduler 任务
- `remove_task(task_id)`：删除 APScheduler 任务
- `pause_task(task)`：暂停任务
- `resume_task(task)`：恢复任务
- `sync_all_active_tasks()`：重载所有有效任务

### 4.2 执行流程

1. APScheduler 触发 `execute_scheduled_task(task_id)`
2. 创建 `ScheduledTaskLog`（running）
3. 执行关联用例（可复用现有 API 执行服务）
4. 更新 `ScheduledTask` 的最近执行信息
5. 写入日志最终状态与明细

### 4.3 Cron 校验

- 在 Serializer 层调用 `CronTrigger.from_crontab(expr)` 校验表达式合法性
- 非法表达式直接返回 400

## 5. API 设计（DRF）

调度任务 CRUD：

- `GET /api/execution/scheduled-tasks/`
- `POST /api/execution/scheduled-tasks/`
- `GET /api/execution/scheduled-tasks/{id}/`
- `PUT/PATCH /api/execution/scheduled-tasks/{id}/`
- `DELETE /api/execution/scheduled-tasks/{id}/`

扩展动作：

- `POST /api/execution/scheduled-tasks/{id}/pause/`
- `POST /api/execution/scheduled-tasks/{id}/resume/`
- `POST /api/execution/scheduled-tasks/{id}/trigger/`

日志查询：

- `GET /api/execution/scheduled-task-logs/`
- `GET /api/execution/scheduled-task-logs/{id}/`

## 6. 异常处理与日志

- 调度核心函数需全程 `try/except`，避免单任务异常影响调度器进程
- 关键路径记录日志：
  - 任务注册/更新/删除
  - 调度触发
  - 执行失败堆栈
- 日志中禁止打印敏感变量明文

## 7. 运行与部署建议

- 开发环境可在 `runserver` 下自动启动调度器
- 生产建议拆分独立调度进程（避免多实例重复触发）
- 多实例部署时需增加分布式锁或仅单实例启用 scheduler

# 25-AI治理（用量配额与审计）开发文档

## 0. 需求来源与开发动因

- **业务价值摘要**：对外可交付场景必须具备 AI 成本可控、滥用可防、问题可追溯三件套：配额/并发、用量统计、审计追踪。
- **业务背景**：平台已具备 AI 生成用例、流式生成、知识库自动填表等能力；一旦进入多团队/多用户使用，必须具备治理能力，否则会出现“成本不可控、偶发故障难定位、合规难过审”。
- **现状痛点**：缺少统一的 AI 调用审计与用量统计；缺少用户级并发保护；缺少管理员侧可视化查询接口。
- **建设目标**：提供最小可用的 AI 治理能力（MVP）：审计落库、配额/并发保护、管理员查询接口。
- **预期收益**：平台上线后可控可查，支持后续扩展到组织级配额、计费与告警。

## 1. 能力范围（本期 MVP）

### 1.1 审计与用量统计（落库）

- 新增审计模型 `assistant.models.AiUsageEvent`
- 对关键 AI 入口写入审计：
  - `POST /api/ai/generate-cases/`
  - `POST /api/ai/generate-cases-stream/`
  - `POST /api/ai/phase1-preview/`
  - `POST /api/ai/test-connection/`
  - `POST /api/ai/verify-connection/`
  - `POST /api/assistant/knowledge/autofill-from-file/`

审计字段包括：

- action、endpoint、user、success、status_code、latency_ms
- model_used/test_type/module_id（可选）
- prompt_chars/output_chars/cases_count
- error_code/error_message（脱敏摘要）
- meta（脱敏后的 JSON）

### 1.2 配额与并发保护（按用户）

在进入 AI 长耗时流程前进行拦截：

- **每日调用配额**（可选启用）：超过阈值返回 `429`
- **并发上限**（默认启用）：超过阈值返回 `429`
- 并发槽位带 TTL，避免异常中断导致长期占用

### 1.3 管理员查询接口

新增系统级接口（仅系统管理员）：

- `GET /api/sys/ai-usage/events/`：查看最近 N 条审计事件（支持筛选）
- `GET /api/sys/ai-usage/summary/`：查看近 N 天按 action 聚合统计
- `GET /api/sys/ai-usage/metrics/`：趋势（按天）+ 耗时指标（avg/max/p50/p95）
- `GET /api/sys/ai-usage/top-errors/`：近 N 天失败原因 TopN（错误码/摘要）
- `GET /api/sys/ai-usage/latency-trend/`：P95 按天趋势（按 action）
- `GET /api/sys/ai-usage/export.csv`：按筛选条件导出 CSV（流式）

## 2. 配置项（环境变量）

配置文件：`AITestProduct/settings.py`

| 变量名 | 默认值 | 说明 |
|------|------|------|
| `AI_GUARD_DAILY_REQUESTS` | `0` | 每用户每日最大请求数；`0` 表示不启用配额 |
| `AI_GUARD_MAX_CONCURRENCY` | `2` | 每用户最大并发（同步+流式合并计数）；`0` 表示不限制（不建议） |
| `AI_GUARD_CONCURRENCY_TTL_SECONDS` | `180` | 并发计数 TTL（秒），用于异常中断自动释放 |

## 3. 代码入口

- 审计模型：`assistant/models.py` → `AiUsageEvent`
- 治理工具：`assistant/services/ai_governance.py`
  - 配额计数：`check_and_increment_daily_quota`
  - 并发槽：`acquire_ai_concurrency_slot` / `release_ai_concurrency_slot`
  - 审计写入：`write_ai_usage_event`（内置脱敏）
- AI 入口写入点：`assistant/views.py`
- 管理员查询接口：`user/sys_views.py`、`user/sys_urls.py`
- 前端页面（管理员）：`frontend/src/views/system/AiUsageDashboard.vue`
- 前端接口封装：`frontend/src/api/sysAiUsage.js`

## 4. 开发过程（记录）

1. **复用现有审计风格**
   - 参考 `server_logs/audit.py` 的“脱敏 + 写入失败不影响主流程”原则。
2. **模型设计**
   - 明确“不存敏感原文”，以字符数、枚举与脱敏摘要为主。
3. **治理策略落地**
   - 在进入 AI 长耗时调用前完成配额与并发校验，失败返回 429。
4. **管理员查询接口**
   - 先提供最小可用的事件列表与按 action 汇总，为后续接入 UI/告警打基础。

## 5. 迁移与上线步骤

1. 执行迁移：

```bash
python manage.py makemigrations assistant
python manage.py migrate
```

注意：当前项目数据库密码已改为环境变量 `DB_PASSWORD` 注入；若未设置会导致连接 MySQL 失败，从而无法执行迁移。

2. （可选）配置生产环境配额：

```bash
export AI_GUARD_DAILY_REQUESTS=200
export AI_GUARD_MAX_CONCURRENCY=2
export AI_GUARD_CONCURRENCY_TTL_SECONDS=180
```

3. 验证接口可用（见下一节验收清单）。

## 6. 验收清单（最小）

- [ ] 调用 `POST /api/ai/generate-cases/` 成功后，`AiUsageEvent` 增加一条 `generate_cases` 事件

- [ ] 触发并发：设置 `AI_GUARD_MAX_CONCURRENCY=1`，同时发起两次请求，其中一次返回 `429`
- [ ] 管理员接口：

- [ ] 前端页面（系统管理员）：
  - [ ] 侧边栏「系统管理」下可进入「AI 用量与审计」

## 7. 回滚方案

1. **配置层回滚（推荐）**
   - 关闭配额：`AI_GUARD_DAILY_REQUESTS=0`
   - 关闭并发限制（不推荐长期）：`AI_GUARD_MAX_CONCURRENCY=0`
2. **代码层回滚**
   - 回退 `assistant/views.py` 中治理拦截与审计写入点
   - 删除 `AiUsageEvent` 相关迁移（需谨慎，建议保留表仅停止写入）

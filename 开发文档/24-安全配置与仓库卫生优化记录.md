# 24-安全配置与仓库卫生优化记录

## 0. 变更背景

目标：把“高风险默认值/明文配置”从代码仓库中移除，并降低无关文件误提交导致的协作噪音与泄露风险。

本次优化聚焦两类问题：

- **安全配置治理**：`SECRET_KEY`、数据库密码、Elasticsearch 密码等不应以可用默认值出现在仓库中。
- **仓库卫生**：IDE 配置、Vite 临时文件、`__pycache__` 等不应进入版本控制。

---

## 1. 变更范围

### 1.1 后端配置（Django）

文件：`AITestProduct/settings.py`

变更点：

- `DEBUG` 改为读取环境变量 `DJANGO_DEBUG`（默认 `True`）
- `SECRET_KEY` 改为读取环境变量 `DJANGO_SECRET_KEY`
  - 开发环境（`DJANGO_DEBUG=1`）：允许缺省，使用临时值以避免阻塞首次启动
  - 生产环境（`DJANGO_DEBUG=0`）：缺省将 **启动失败**
- 数据库配置改为读取 `DB_NAME/DB_USER/DB_PASSWORD/DB_HOST/DB_PORT`
  - 生产环境（`DJANGO_DEBUG=0`）：`DB_PASSWORD` 缺失将 **启动失败**
- Elasticsearch 密码改为仅读取环境变量 `ELASTICSEARCH_PASSWORD`（不再提供可用默认值）
  - 生产环境（`DJANGO_DEBUG=0`）：若设置了 `ELASTICSEARCH_USER`，则 `ELASTICSEARCH_PASSWORD` 缺失将 **启动失败**

### 1.2 仓库忽略规则（gitignore）

文件：`.gitignore`

变更点：

- 忽略 `.idea/`、`.vscode/`
- 忽略 `frontend/vite.config.js.timestamp-*.mjs`（Vite 热更新产生的临时文件）

## 2. 开发过程（可复盘）

1. **现状盘点**
   - 发现 `settings.py` 存在可用的 `SECRET_KEY`、`DB_PASSWORD`、`ELASTICSEARCH_PASSWORD` 明文默认值。
   - `git status` 中出现 `.idea/workspace.xml`、`__pycache__`、Vite `timestamp-*.mjs` 等无关文件。

2. **安全策略确定**
   - 开发体验优先：开发环境允许缺省 key/密码（便于首次启动与本地调试）。
   - 生产安全优先：生产模式下缺关键变量直接失败，避免“带着不安全默认值运行”。

3. **实现与自检**
   - 在 `settings.py` 增加布尔环境变量解析 `_env_bool`
   - 将密钥/密码改为环境变量读取，并增加生产模式下的启动强校验
   - 扩展 `.gitignore` 覆盖 IDE 与 Vite 临时文件

4. **文档同步**
   - 更新 `06-部署与配置指南.md`：加入“生产安全启动校验”说明
   - 更新 `07-安全与权限基线.md`：补充外部依赖凭据治理与强校验规则

## 3. 迁移步骤（开发 / 测试 / 生产）

### 3.1 开发环境（本地）

可不设置 `DJANGO_SECRET_KEY/DB_PASSWORD` 也能启动（仅用于调试）。推荐仍通过环境变量显式设置，便于团队一致性。

PowerShell 示例：

```powershell
$env:DJANGO_DEBUG="1"
$env:DJANGO_SECRET_KEY="dev-secret"
$env:DB_NAME="ai_test_product"
$env:DB_USER="root"
$env:DB_PASSWORD="your_password"
$env:DB_HOST="127.0.0.1"
$env:DB_PORT="3306"
python manage.py runserver
```

### 3.2 生产/类生产环境

必须设置：

- `DJANGO_DEBUG=0`
- `DJANGO_SECRET_KEY`
- `DB_PASSWORD`（以及 DB 相关连接项）
- （如启用 ES 且设置了 `ELASTICSEARCH_USER`）必须设置 `ELASTICSEARCH_PASSWORD`

## 4. 验证方式（建议最小集合）

- **启动校验**
  - `DJANGO_DEBUG=0` 且不设置 `DJANGO_SECRET_KEY` → 应启动失败并报出明确错误
  - `DJANGO_DEBUG=0` 且不设置 `DB_PASSWORD` → 应启动失败并报出明确错误
- **基本链路**
  - 能正常启动后端、迁移通过、登录接口可用
- **日志模块（如启用 ES）**
  - 配置 ES 账号密码后能正常初始化相关 client（若有）

---

## 5. 回滚方案

若本次变更导致环境启动被阻断，优先按以下顺序处理：

1. **补齐环境变量**：这是推荐路径（更安全）
2. 临时回退到开发模式：设置 `DJANGO_DEBUG=1`（仅用于应急定位，不建议长期使用）
3. 代码层回滚：回退 `settings.py` 与 `.gitignore`（不推荐，除非紧急止血）

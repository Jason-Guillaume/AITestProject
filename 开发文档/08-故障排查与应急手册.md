# 08-故障排查与应急手册

## 0. 需求来源与开发动因

- 业务价值摘要：固化应急处置流程，提升故障定位和恢复效率。
- 业务背景：功能链路持续延长（执行引擎、AI、RAG、调度、环境检查），故障类型与影响面不断扩大。
- 现状痛点：缺少统一排查手册时，故障处置依赖个人经验，定位慢、止血慢、复盘难。
- 建设目标：形成标准化应急流程与分级策略，明确处置路径和责任边界。
- 预期收益：提升故障响应效率，沉淀可复用的排障与复盘经验。

## 1. 目标与适用场景

本手册用于在系统出现异常时提供统一的排查路径与应急流程，适用于：

- 本地开发联调故障
- 测试环境稳定性问题
- 生产环境接口不可用、性能退化、AI 服务异常等故障

---

## 2. 故障分级建议

| 等级 | 影响范围 | 示例 | 处置目标 |
|------|------|------|------|
| P1 | 核心业务不可用 | 登录失败、接口全量 5xx | 15 分钟内止血，1 小时内恢复 |
| P2 | 主要功能受限 | AI 生成不可用、性能任务异常 | 30 分钟内定位，4 小时内恢复 |
| P3 | 局部或可绕过问题 | 单页面异常、个别接口超时 | 当日修复或纳入迭代 |

---

## 3. 通用排查流程

```
收到告警/用户反馈
  ↓
确认影响范围（用户数、接口数、业务链路）
  ↓
快速判断：前端问题 / 网关问题 / 后端问题 / 外部依赖问题
  ↓
查看关键日志与指标（应用日志、Nginx、DB、Redis）
  ↓
执行临时止血（限流、降级、回滚、切换）
  ↓
恢复后做根因分析与复盘
```

## 4. 高频故障与处理指引

### 4.1 登录失败（401/403）

**现象**：

- 登录后页面反复跳转到 `/login`
- 所有业务请求返回 401

**排查步骤**：

1. 检查 `/api/user/login/` 是否返回 token
2. 检查前端是否写入 `localStorage.token`
3. 检查请求头是否为 `Authorization: Token <token>`
4. 检查后端 DRF 认证配置是否被改动

**临时处置**：

- 清理本地 token 后重新登录
- 回滚最近的认证相关改动

### 4.2 接口 404 / 路由不通

**现象**：

- 前端页面正常打开，但 API 返回 404

**排查步骤**：

1. 检查代理与网关：`/api` 是否正确转发
2. 检查 Django 主路由 `AITestProduct/urls.py` 是否包含目标前缀
3. 检查接口是否缺少尾斜杠（DRF 默认风格）

**临时处置**：

- 修正网关路由规则并热更新
- 前端统一补全尾斜杠

### 4.3 数据库连接异常

**现象**：

- 启动报 `OperationalError`
- 接口大量 500

**排查步骤**：

1. 检查 MySQL 服务状态与端口
2. 核对数据库连接配置（主机、账号、密码、库名）
3. 检查迁移状态：`python manage.py migrate`

**临时处置**：

- 切换到可用数据库实例
- 执行只读降级（如业务允许）

### 4.4 Redis 异常与验证码失败

**现象**：

- 验证码接口异常
- 缓存命中率骤降

**排查步骤**：

1. 检查 Redis 连通性与内存状态
2. 确认是否触发 LocMem 降级
3. 检查多实例场景下验证码一致性问题

**临时处置**：

- 恢复 Redis 服务
- 单实例临时承载验证码流程

### 4.5 AI 生成失败/超时

**现象**：

- `/api/ai/test-connection/` 返回 4xx/5xx
- 流式生成中断或长时间无响应

**排查步骤**：

1. 检查 `/api/sys/ai-config/` 是否配置正确
2. 校验 API Key 与模型地址可用性
3. 检查代理超时与网络连通
4. 检查上游模型平台状态

**临时处置**：

- 切换备用模型或备用 base_url
- 临时关闭流式，改用同步生成接口

### 4.6 本地开发：端口占用/服务“假死”（8000 LISTEN 但请求超时）

**现象**：

- `127.0.0.1:8000` 显示在监听（LISTENING），但 `/api/user/captcha/`、`/api/user/login/` 请求超时或间歇性 5xx
- 或同一个项目启动了多个 `runserver/daphne`，出现端口争用与反复重载

**排查步骤**：

1. 查看端口占用 PID：

```powershell
netstat -ano | findstr ":8000" | findstr LISTENING
```

2. 查看该 PID 的命令行是否属于本项目：

```powershell
(Get-CimInstance Win32_Process -Filter "ProcessId=<PID>").CommandLine
```

3. 若确认是本项目残留进程，结束该 PID 并重新启动。

**推荐处置**：

- 使用仓库自带脚本 `start-dev.ps1` 启动，它包含“清理残留 runserver 进程 + 兜底按端口 8000 清理”的预处理，能显著减少端口占用/假死。

### 4.7 本地开发：WebSocket 高并发 + 自动重载导致全站 500

**现象**：

- 多个 WebSocket 连接存在时触发代码热更新（dev reload），Daphne 日志出现：
  - `took too long to shut down and was killed`
  - `cannot schedule new futures after interpreter shutdown`
- 随后 `/captcha`、`/login` 等普通 HTTP 也开始 500

**根因**（常见）：

- WS 生命周期不收敛：SSH 线程结束/报错后连接未主动关闭，导致 reload 时残留任务被强杀，进程进入半退出状态。

**处置与修复建议**：

- 优先升级到当前仓库实现：日志模块 Consumer 在 `error/done` 时会主动 `close()`，并将 SSH connect 超时缩短用于“快速失败”，降低 reload 残留。
- 若线上环境也存在类似问题，建议将 ASGI 进程从 `runserver` 切换到稳定的进程托管（systemd/supervisor）并配合反向代理健康检查。

## 5. 运行与诊断命令清单

### 5.1 基础健康检查

```bash
# Django 迁移检查
python manage.py migrate --plan

# 启动服务验证
python manage.py runserver
```

### 5.2 测试快速回归

```bash
# 仅跑冒烟，快速判断核心链路
pytest -m smoke tests/api
```

### 5.3 用例编号一致性检查

```bash
python manage.py resequence_case_numbers --dry-run --strict --report-file reports/case-number-check.txt --report-json-file reports/case-number-check.json
```

## 6. 应急止血策略

### 6.1 回滚策略

建议保留最近两个可运行版本，故障时按以下顺序回滚：

1. 前端静态资源版本回滚
2. 后端应用版本回滚
3. 配置回滚（环境变量、网关规则）
4. 数据库变更回滚（需谨慎，优先做兼容迁移）

### 6.2 降级策略

- AI 功能降级：隐藏入口或提示稍后重试
- 非核心统计接口降级：返回缓存结果或默认空值
- 可写操作降级：临时只读模式（按业务风险评估）

## 7. 故障复盘模板（建议）

每次 P1/P2 故障建议记录：

1. 发生时间与恢复时间
2. 影响范围（用户、功能、时长）
3. 根因分析（代码、配置、外部依赖、流程）
4. 临时止血动作
5. 长期修复项（责任人 + 截止时间）
6. 预防机制（监控、测试、流程改进）

## 8. 常见场景速查表

| 现象 | 优先检查 |
|------|------|
| 全站接口 401 | token 注入、登录接口返回、DRF 认证配置 |
| 页面可开但接口 404 | 网关 `/api` 转发、后端路由注册 |
| 上传图片失败 | `MEDIA_ROOT`、网关 `/media`、文件大小/后缀校验 |
| AI 流式断流 | 上游模型状态、代理超时、SSE 连接稳定性 |
| 性能任务无法执行 | 任务状态是否 `running/completed`、`task_id` 是否正确 |

# 05-测试指南与性能评测

## 0. 需求来源与开发动因

- 业务价值摘要：统一测试方法与质量口径，提升版本发布可靠性。
- 业务背景：功能和模块持续增多，需要统一测试策略与执行标准保障迭代质量。
- 现状痛点：缺少统一测试入口时，覆盖不均、回归遗漏与性能问题后置暴露较常见。
- 建设目标：沉淀“如何测、测什么、如何评估”的统一规范。
- 预期收益：形成可复用质量保障闭环，提升版本发布稳定性。

## 1. 测试体系概览

当前项目的测试体系由三层组成：

1. **后端单元/模块测试（Django TestCase）**
   - 各业务应用均保留 `tests.py` 入口，当前主要已有实测内容集中在 `assistant/tests.py`。

2. **后端 API 集成测试（pytest + requests）**
   - 位于 `tests/api/`，覆盖鉴权、核心业务链路、边界输入、错误方法等。
   - 通过环境变量注入测试账号与可选 AI Key。

3. **CI 自动化执行（GitHub Actions）**
   - 工作流 `.github/workflows/api-tests.yml` 自动拉起 MySQL/Redis，执行迁移、数据校验和冒烟用例。

### 2.2 pytest 配置说明

`pytest.ini` 当前约定：

- `testpaths = tests`
- `python_files = test_*.py`
- `python_classes = Test*`
- `python_functions = test_*`
- marker 分层：
  - `smoke`：核心链路冒烟
  - `regression`：回归验证
  - `full`：全量矩阵

## 3. 本地测试环境准备

### 3.1 安装依赖

```bash
pip install -r requirements.txt
pip install -r requirements-dev.txt
```

开发测试依赖（`requirements-dev.txt`）包括：

- `pytest`
- `requests`
- `pytest-html`
- `allure-pytest`

### 3.2 启动被测服务

```bash
python manage.py migrate
python manage.py runserver
```

默认测试目标地址：`http://127.0.0.1:8000/api`

### 3.3 必要环境变量

API 集成测试通过 `tests/api/conftest.py` 读取以下环境变量：

| 变量名 | 必填 | 说明 |
|------|------|------|
| `TEST_API_BASE_URL` | 否 | API 根地址，默认 `http://127.0.0.1:8000/api` |
| `TEST_API_USERNAME` | 是 | 普通测试账号 |
| `TEST_API_PASSWORD` | 是 | 普通测试账号密码 |
| `TEST_API_ADMIN_USERNAME` | 否 | 管理员账号（缺省时回退普通账号） |
| `TEST_API_ADMIN_PASSWORD` | 否 | 管理员密码 |
| `TEST_AI_API_KEY` | 否 | AI 连通性 happy-path 用例可选启用 |

若缺少必填变量，测试会以 `pytest.skip` 跳过对应套件，而不是硬失败。

## 4. 测试脚本结构说明

### 4.1 `tests/api/conftest.py`（夹具层）

核心职责：

1. **统一登录**
   - 通过 `/api/user/login/` 获取用户 token 与管理员 token。

2. **构建认证客户端**
   - 为后续请求注入 `Authorization: Token <token>`。

3. **构造业务种子对象**
   - 自动创建项目、版本、模块、用例、步骤、测试计划、报告、性能任务、缺陷等对象。
   - 为回归用例提供可复用的实体 ID（`seed_objects`）。

### 4.2 `tests/api/test_backend_api_suite.py`（用例层）

覆盖内容包括：

- 公共接口免鉴权验证（captcha/register/login）
- 受保护接口鉴权拦截验证（401/403）
- 用户、项目、用例、执行、缺陷、AI 等主域接口的 happy path
- 参数非法场景（类型错误、枚举越界、缺字段）
- 错误 HTTP Method 场景（401/403/405）

标记策略：

- `@pytest.mark.smoke`：关键链路可用性
- `@pytest.mark.regression`：中等覆盖回归
- `pytestmark = [pytest.mark.full]`：文件级全量标记

### 4.3 `assistant/tests.py`（AI 单元测试）

该文件是当前最完整的单元测试集合，重点覆盖：

- 模型路由与 URL 归一化逻辑
- 提示词构建与分发策略
- API 用例 AI 生成字段清洗与补齐
- `AiTestConnectionAPIView` 的关键行为（Mock OpenAI）

## 5. 如何运行测试

### 5.1 运行全部 API 集成测试

```bash
pytest tests/api
```

### 5.2 按 marker 运行

```bash
# 冒烟测试
pytest -m smoke tests/api

# 回归测试
pytest -m regression tests/api

# 全量测试
pytest -m full tests/api
```

### 5.3 生成 HTML 报告

```bash
pytest -m smoke tests/api --html=reports/api-test-report.html --self-contained-html
```

报告路径：`reports/api-test-report.html`

### 5.4 运行 Django 单元测试

```bash
python manage.py test assistant
python manage.py test
```

## 6. CI 流水线测试机制

`api-tests.yml` 的关键步骤如下：

1. 检出代码，安装 Python 3.11
2. 启动 MySQL 8 与 Redis 7 服务容器
3. 安装 `requirements.txt` 与 `requirements-dev.txt`
4. 执行 `python manage.py migrate --noinput`
5. 执行编号一致性检查（dry-run）：
   - `python manage.py resequence_case_numbers --dry-run --strict ...`
6. 启动 Django 服务
7. 执行冒烟集成测试并上传 HTML 报告
8. 上传编号检查报告（txt/json）

该流程可保证“数据库迁移 + 核心 API 可用 + 用例编号规则”在 PR/主干上持续健康。

## 7. 性能评测能力与现状

### 7.1 性能任务模型（PerfTask）

系统通过 `PerfTask` 管理压测任务元数据：

- 场景：`jmeter` / `locust`
- 并发：`concurrency`
- 持续时间：`duration`（如 `30s`、`10m`、`2h`）
- 状态机：`pending -> running -> completed/failed`

对应接口：

- `/api/perf/tasks/`（列表/创建）
- `/api/perf/tasks/{task_id}/`（详情/删除）
- `/api/perf/tasks/{task_id}/run/`（触发运行）

### 7.2 评测数据来源

当前版本的“性能评测”主要用于**任务管理与状态流转**，并不直接执行真实 JMeter/Locust 脚本。

现有可观测数据：

- 性能任务状态与执行人
- Dashboard 聚合接口中的活动流（运行中任务）
- 测试报告/缺陷/执行数量的趋势统计（业务层性能指标）

### 7.3 演示与压测准备

可通过管理命令注入演示任务：

```bash
python manage.py seed_perf_tasks
```

命令行为：

- 若库中已有未删除任务，则跳过
- 若为空，插入 6 条示例任务（覆盖 running/completed/failed）

## 8. 如何扩展现有测试用例

### 8.1 新增 API 集成用例（推荐）

步骤建议：

1. 在 `tests/api/test_backend_api_suite.py` 新增 `test_` 函数
2. 优先复用 `conftest.py` 的 `authed_client`、`admin_client`、`seed_objects`
3. 同时覆盖：
   - happy path（200/201/204）
   - 非法输入（400）
   - 鉴权/权限（401/403）
4. 为用例打上合适 marker（`smoke`/`regression`/`full`）

### 8.2 为新模块补齐 Django 单元测试

当前 `user/project/testcase/execution/defect/common` 下 `tests.py` 多为占位文件。建议按以下策略补齐：

- 模型层：字段约束、状态流转、软删除行为
- 序列化层：边界校验与错误消息
- 视图层：权限与动作分支（`@action`）
- 服务层：纯函数/无副作用逻辑优先单测

### 8.3 扩展 AI 相关测试

建议延续 `assistant/tests.py` 的做法：

- 对外部依赖（OpenAI SDK、网络）使用 `unittest.mock`
- 固定输入输出，避免模型随机性导致 CI 抖动
- 将“提示词逻辑”和“结果归一化逻辑”单独测试

## 9. 常见问题与排查

| 问题 | 现象 | 处理建议 |
|------|------|------|
| 缺少测试账号环境变量 | 用例被 skip | 设置 `TEST_API_USERNAME/TEST_API_PASSWORD` |
| 数据库未迁移 | 关键接口 500 或表不存在 | 先执行 `python manage.py migrate` |
| 401 大量出现 | 受保护接口全部失败 | 检查登录接口是否可用、token 是否注入 |
| AI 用例不稳定 | 连通性相关用例偶发失败 | 使用 Mock 或将 AI key 相关用例设为可选 |
| 性能任务 run 返回 400 | 已 running/completed 不能重复触发 | 按状态机规则重建或复制任务 |

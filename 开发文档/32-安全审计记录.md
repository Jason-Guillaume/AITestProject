# 32-安全审计记录

## 0. 文档目的

- 记录对仓库进行的 **依赖漏洞扫描** 与 **静态安全扫描（Bandit）** 结果，便于版本对比与复测。
- 说明 **Django 部署检查** 在本项目中的使用前提（生产环境变量），避免误读「未执行」为「无问题」。

本文 **不替代** 渗透测试与业务逻辑审计；建议按发布节奏定期更新「变更记录」并重新执行下方命令。

## 2. 执行命令（复测用）

在项目根目录（后端）与 `frontend` 目录执行：

```bash
# Python 依赖（需在 venv 中安装 pip-audit）
python -m pip_audit -r requirements.txt

# 前端依赖
cd frontend && npm audit

# Bandit（需在 venv 中安装 bandit）
python -m bandit -r AITestProduct assistant common defect execution project server_logs testcase user -x "**/migrations/**" -ll

# Django 部署项（需完整生产向环境变量，见 §4）
set DJANGO_DEBUG=0
set DJANGO_SECRET_KEY=<足够长度的密钥>
set DB_PASSWORD=<非空数据库密码>
python manage.py check --deploy
```

PowerShell 可将 `set` 换为 `$env:NAME="value"`。

## 3. 审计结果快照

**快照日期：2026-04-16**（与下方变更记录一致时可合并更新）。

### 3.1 Python 依赖（pip-audit）

| 结果 | 说明 |
|------|------|
| **未发现已知漏洞** | 对 `requirements.txt` 解析后与 PyPI 漏洞库比对，无匹配 CVE |

### 3.2 前端依赖（npm audit）

| 严重程度 | 数量 | 说明 |
|----------|------|------|
| Moderate | **2** | **esbuild ≤0.24.2**（[GHSA-67mh-4wv8-2f99](https://github.com/advisories/GHSA-67mh-4wv8-2f99)）：开发服务器场景下，恶意站点可向 dev server 发起请求并读取响应；经 **Vite 5.x** 依赖链引入。 |

**处置建议**：

- **生产构建**：`vite build` 产物不运行 Vite dev server，**终端用户面风险显著低于开发机**。
- **开发机**：勿将 `npm run dev` 暴露于公网；仅内网或本机访问。
- **升级**：`npm audit fix --force` 可能将 Vite 升至 **8.x**，属 **破坏性变更**，需单独排期兼容与回归。

### 3.3 静态代码（Bandit，`-ll` 起报 Medium+）

| 等级 | 条数（概览） | 位置与说明 |
|------|----------------|------------|
| **High** | 1 | `server_logs/ssh_tail.py`：**`paramiko.AutoAddPolicy()`**（**`SERVER_LOGS_SSH_HOST_KEY_POLICY=auto_add` 分支**）— 未知主机指纹自动信任，存在 **MITM** 风险。**生产默认已改为 `reject`（系统 known_hosts + RejectPolicy）**，见 `开发文档/33` §3.1；静态扫描仍可能报该行，属 **显式开发策略** 残留。 |
| **Medium** | 2 | `assistant/tasks.py`：**B310** `urllib.request.urlopen` — 若 URL 来自不可信输入，需限制 scheme 为 **http(s)**。`server_logs/ssh_tail.py`：**B601** `exec_command(cmd)` — 需保证 `log_path` 等经既有 **validators** 净化并持续审计调用链。 |
| **Low** | 若干 | 多为通用启发式规则；可按模块评估后 `# nosec Bxxx` 或收窄扫描路径。 |

### 3.4 Django `check --deploy`

| 状态 | 说明 |
|------|------|
| **CI 已纳入（首期）** | `.github/workflows/api-tests.yml` 在 migrate 后执行 **`python manage.py check --deploy`**，并注入满足 `settings.py` 生产校验的 **最小环境变量**（`DJANGO_SECRET_KEY`、`DB_*`、`ELASTICSEARCH_*` 等）。 |
| **本地/未配齐变量** | 项目在 `DJANGO_DEBUG=0` 时要求 **`DB_PASSWORD` 非空** 等，未配齐时进程在 **加载 settings 阶段退出**，属 **预期安全设计**。 |

**建议**：预发环境继续用与线上一致的变量集跑 `check --deploy`，逐项处理新增的 `SECURE_*`、`CSRF_COOKIE_SECURE` 等告警并在本文件或 `33` 备案。

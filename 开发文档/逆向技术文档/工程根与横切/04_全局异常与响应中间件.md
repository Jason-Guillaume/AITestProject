# 全局异常与响应中间件

## 文档目的与读者

本文说明 **`GlobalExceptionMiddleware`** 的设计意图、**返回 JSON 形状**，以及当前仓库 **是否在 `MIDDLEWARE` 中启用**。适合判断「接口报错格式为何与文档不一致」、以及评估是否开启全局捕获。

---

## 1. 实现位置与职责

**文件**：`common/middleware/exception_middleware.py`

- 包装 Django `get_response`：正常返回原响应；**任意未捕获异常**进入 `handle_exception`。
- 分类处理：
  1. **`BusinessException`**（`common.exceptions`）→ 使用异常的 `status_code` 与 `detail`。
  2. **DRF `APIException`** → 同上。
  3. **其它 `Exception`** → HTTP **500**，`message` 固定为 **「服务器内部错误」**，并 **`logger.exception`** 记录堆栈。

成功路径的 **`JsonResponse` 字段**（三类异常结构一致）：

- `code`：HTTP 状态码数值（与 `status` 参数一致）。
- `message`：字符串化 `detail` 或固定中文提示。
- `data`：恒为 `null`。
- `timestamp`：`timezone.now().isoformat()`。

---

## 2. 与 `settings.MIDDLEWARE` 的当前关系（重要）

`AITestProduct/settings.py` 的 **`MIDDLEWARE` 列表中默认未注册** `GlobalExceptionMiddleware`。

因此：

- 多数视图仍走 **DRF 默认异常处理器** 或各视图手写 `Response`，响应体可能是 DRF 标准 `{"detail": ...}` 或项目封装的 **`ApiResponse`**（见 `common/response.py`），**不一定**含 `timestamp`。
- 若将本中间件加入 `MIDDLEWARE`，需评估：
  - 与 **`REST_FRAMEWORK['EXCEPTION_HANDLER']`**（若自定义）的先后顺序；
  - **`StreamingResponse`**、文件下载等是否被 `JsonResponse` 误包（中间件只捕获 `get_response` 抛出的异常，不改编正常流式响应，但若视图内部生成器抛错会被转成 JSON 500）。

**启用方式**（摘自中间件文件注释）：在 `MIDDLEWARE` 列表中加入：

`'common.middleware.exception_middleware.GlobalExceptionMiddleware'`

建议放在 **靠后** 位置，以便尽量让业务中间件先处理请求。

## 3. 日志级别

| 异常类型 | 日志方法 | 说明 |
|----------|-----------|------|
| Business / APIException | `logger.warning` | 带 `path`、`username` |
| 未知异常 | `logger.exception` | 含完整 traceback |

**不向客户端返回** 未知异常的消息体或堆栈，避免信息泄漏。

## 5. 与 `ApiResponse` 的关系（概念）

业务成功路径常使用 `common/response.py` 的 **`ApiResponse`** 统一 `code/message/data`；本中间件在 **异常路径** 上提供了 **相似字段名** 的 JSON，便于前端统一解析——**但仅当中间件实际启用时生效**。

---

## 6. 显式与静默

| 显式 | 静默 |
|------|------|
| 客户端收到的 `code`、`message`、`timestamp` | 500 的真实异常类型与堆栈仅服务端日志 |
| WARNING 级业务/API 日志 | 中间件未启用时，本文件描述的行为 **不发生** |

---

## 7. 排障

1. **期望统一 JSON 但未生效**：先 `grep MIDDLEWARE GlobalExceptionMiddleware`。
2. **启用后部分接口格式仍不同**：该接口可能在 `try/except` 内吞掉异常并自行 `Response`。
3. **Streaming 中断**：检查是否在生成器外未捕获异常导致进入中间件。

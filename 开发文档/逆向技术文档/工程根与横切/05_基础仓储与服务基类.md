# 基础仓储与服务基类

## 文档目的与读者

本文描述 **`BaseRepository`** 与 **`BaseService`**：在部分业务模块中作为 **View 之下的第二层**，封装缓存、事务、软删与生命周期钩子。适合理解「为何 `get_by_id` 带缓存」或扩展 **Service 层审计** 时参考。

---

## 1. 文件与职责边界

| 类型 | 路径 | 职责 |
|------|------|------|
| 仓储 | `common/repositories/base_repository.py` | 模型级 CRUD、默认 `is_deleted=False`、可选 Django Cache |
| 服务 | `common/services/base_service.py` | 委托仓储；`create`/`update` 注入 `creator`/`updater`；钩子 `_after_*` |

二者 **不注册 HTTP 路由**；由具体 `XxxService` + `APIView`/`ViewSet` 组合对外。

---

## 2. BaseRepository 要点

### 2.1 初始化

- `self.model`：ORM 模型类。
- `self.cache_prefix`：默认 **`model._meta.db_table`**，用作缓存键前缀。
- `self.cache_ttl`：默认 **300** 秒。

### 2.2 `get_by_id(obj_id)`

1. 键 **`{db_table}:{obj_id}`** 尝试 `cache.get`。
2. 未命中则 `model.objects.filter(id=obj_id, is_deleted=False).first()`，命中则 `cache.set`。
3. 不存在返回 **`None`**（调用方必须判空）。

### 2.3 写操作与事务

下列方法均带 **`@transaction.atomic`**（以源码为准）：

- **`create`**：`objects.create`，成功后 **`_invalidate_cache()`**（无参：尝试 `delete_pattern` 清前缀）。
- **`bulk_create`**：批量插入后同样失效缓存。
- **`update(obj_id, data)`**：先 `get_by_id`；逐字段 `setattr` + `save`；**`_invalidate_cache(obj_id)`**。
- **`update_by_filter`**：批量 `update` 后 `_invalidate_cache()`。
- **`delete(obj_id, soft=True)`**：默认软删 `is_deleted=True`；`soft=False` 时物理 `delete()`；最后 **`_invalidate_cache(obj_id)`**。
- **`delete_by_filter`**：按条件软删或硬删，再 `_invalidate_cache()`。

### 2.4 只读查询

- **`get_all(filters)`**：`is_deleted=False` 上叠加 `**filters`。
- **`get_by_ids`**、`exists`、`count`：均基于未删除集。

### 2.5 缓存失效 `_invalidate_cache(obj_id=None)`

- 传入 **`obj_id`**：删除单键 `{table}:{id}`。
- **未传**：若 `cache.delete_pattern` 存在则 `delete_pattern(f"{prefix}:*")`；否则静默跳过（注释说明 **LocMemCache** 测试环境可能不支持 `delete_pattern`）。

## 3. BaseService 要点

### 3.1 构造

`def __init__(self, repository: BaseRepository[T])`，子类典型写法：`super().__init__(XxxRepository())`。

### 3.2 `create(data, user=None)`

- 若 `user` 存在：`data['creator'] = user`（注意与模型字段名一致）。
- 调用 `repository.create(data)` → **`_after_create(obj, user)`**（默认可覆盖）。

### 3.3 `update(obj_id, data, user=None)`

- 若 `user`：`data['updater'] = user`。
- `repository.update` 成功则 **`_after_update(obj, user)`**。

### 3.4 `delete(obj_id, user=None, soft=True)`

- `repository.delete` 成功则 **`_after_delete(obj_id, user)`**。

### 3.5 `bulk_create` / `batch_delete`

- 批量创建时为每条 `data` 注入同一 `creator`；结束后 **`_after_bulk_create`**。
- `batch_delete` 循环调用单条 `delete`，统计成功数。

### 3.6 可覆盖钩子（默认 `pass`）

- **`_after_create`** / **`_after_bulk_create`** / **`_after_update`** / **`_after_delete`**
- **`_validate_create_data`** / **`_validate_update_data`**：预留给子类做校验（基类未自动调用，需子类在 `create`/`update` 前显式使用）。

## 4. 与 BaseModelViewSet 的关系

| 维度 | BaseModelViewSet | BaseRepository + BaseService |
|------|------------------|--------------------------------|
| 入口 | HTTP / DRF | Python 内部调用 |
| 软删 | `perform_destroy` | `repository.delete(soft=True)` |
| 审计 | `record_audit_event` 在 `perform_*` | 可在 `_after_*` 中补写（需自行调用，避免重复） |
| 数据范围 | `get_queryset` 成员过滤 | 仓储默认 **不** 做项目成员过滤，需在 Service 或上层 QuerySet 补 |

同一业务可 **只用 ViewSet**，或 **ViewSet 薄 + Service 厚**；文档需与具体 app 实现一致。

## 6. 显式与静默

| 显式 | 静默 |
|------|------|
| API 返回的业务字段 | 缓存命中/未命中；`delete_pattern` 是否可用 |
| 事务提交成功 | 钩子内异常若未处理可能影响外层事务（子类需谨慎） |

---

## 7. 异常与边界

- **`get_by_id` 返回 `None`**：`update`/`delete` 返回 `None`/`False`，调用链需分支处理。
- **缓存中的 ORM 实例**：若跨进程反序列化失败，依赖 Django 版本与 pickle 策略；生产常用 Redis JSON 序列化需注意 **不可 pickle 的模型实例** 问题（当前实现存的是 ORM 对象，换缓存后端时需评估）。

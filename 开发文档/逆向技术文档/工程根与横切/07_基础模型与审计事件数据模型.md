# 基础模型与审计事件数据模型

## 文档目的与读者

## 1. BaseModel（抽象基类）

**文件**：`common/models.py`  
**`Meta.abstract = True`** — Django **不为** `BaseModel` 单独建表；每个子模型表自带下列列。

### 1.1 字段说明

| 字段 | Django 类型 | 说明 |
|------|----------------|------|
| `creator` | `ForeignKey(AUTH_USER_MODEL, SET_NULL, null=True, blank=True)` | `related_name="%(class)s_created"` |
| `updater` | 同上 | `related_name="%(class)s_updated"` |
| `create_time` | `DateTimeField(auto_now_add=True)` | 创建时间 |
| `update_time` | `DateTimeField(auto_now=True)` | 最后修改时间 |
| `is_deleted` | `BooleanField(default=False)` | **逻辑删除**标记；`BaseModelViewSet.get_queryset` 默认过滤 `False` |

### 1.2 设计意图

- **统一审计维度**：多数业务表可回答「谁创建、谁最后改、何时改」。
- **软删优先**：与物理 `DELETE` 区分，便于恢复与合规保留（是否暴露恢复接口由业务决定）。

### 1.3 前端展示习惯

- 列表常展示 **`create_time`**，可选 **`creator`** 的外链姓名（序列化器 `SerializerMethodField` 或 `StringRelatedField`）。
- **`is_deleted`**、**`updater_id`** 通常不直接展示给普通用户。

## 2. AuditEvent（具体模型）

同样继承 **`BaseModel`**，因此除下表「专有字段」外，仍具备 `creator`、`updater`、`create_time`、`update_time`、`is_deleted`。

### 2.1 动作常量

```48:60:common/models.py
    ACTION_CREATE = "create"
    ACTION_UPDATE = "update"
    ACTION_DELETE = "delete"
    ACTION_EXPORT = "export"
    ACTION_EXECUTE = "execute"

    ACTION_CHOICES = [
        (ACTION_CREATE, "创建"),
```

数据库层 `action` 为 **`CharField(max_length=16, choices=ACTION_CHOICES)`**。

### 2.2 专有字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `object_app` | `CharField(64)` | 对象所在 Django app |
| `object_model` | `CharField(64)` | 小写模型名 |
| `object_id` | `CharField(64)` | 主键字符串形式 |
| `object_repr` | `CharField(255)` | `str(instance)` 截断 |
| `request_path` | `CharField(512)` | HTTP 请求 path |
| `ip` | `CharField(64)` | 客户端 IP（优先 `X-Forwarded-For`） |
| `user_agent` | `CharField(512)` | UA |
| `before` / `after` | `JSONField(null=True)` | 变更前后快照 |
| `extra` | `JSONField(default=dict)` | 扩展 |

### 2.3 Meta

- **`db_table = "audit_event"`**
- **`ordering = ("-create_time",)`** — 默认新在前。
- **索引**：
  - `Index(fields=["action", "-create_time"])`
  - `Index(fields=["object_app", "object_model", "object_id"])`

## 3. 生命周期与写入时机

- **业务 `POST/PATCH/DELETE`** 成功后，审计在服务端 **`perform_*` 或 `record_*` 同步写入**，前端 **无需** 第二次请求「写审计」。
- **读取审计**：必须调用 **独立列表接口**（个人或系统），见 **`06`** 文档第七节。

---

## 4. 与迁移、ORM 的注意点

- 修改 `AuditEvent` 字段后需 **`makemigrations`**；大表加索引注意锁表窗口。
- **`before`/`after` JSON** 若存历史大字段，单表体积增长快，可规划 **归档或异步压缩**（当前代码未实现，属运维扩展）。

## 5. 显式与静默归纳

| 用户界面常显 | 常不显或仅管理员 |

# 组织视图集与多对多关系

## 文档目的

说明 **`Organization`** 模型、**`OrganizationViewSet`** 的 HTTP 能力，以及 **`Organization.members` / `Organization.projects`** 变更时如何通过 **Signal** 向 **`TestProject.members`** 同步。便于理解「入组织即入项目」与 **同步边界**（仅追加、不自动剔除）。

---

## 1. 数据模型 `Organization`

**表**：`sys_org`（`user/models.py`）

| 字段 | 说明 |
|------|------|
| `org_name` | 组织名称，`CharField(max_length=255)` |
| `description` | 可空长文本 |
| `members` | **M2M** → `AUTH_USER_MODEL`，`related_name="organization_memberships"`；用于组织维度权限与 **服务器日志** 等资源共享说明（见模型 `help_text`） |
| `projects` | **M2M** → `project.TestProject`，`related_name="organizations"` |
| 继承 `BaseModel` | `creator`、`updater`、`create_time`、`update_time`、`is_deleted` |

## 2. HTTP：`OrganizationViewSet`

**路由前缀**：`/api/user/orgs/`（`user/urls.py` → `DefaultRouter.register("orgs", ...)`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 列表 | GET | 分页 `SystemPagination`：`page_size` 默认 10，`page_size_query_param`，`max_page_size=100` |
| 详情/创建/更新/删除 | 标准 `ModelViewSet` | 删除走 **`BaseModelViewSet.perform_destroy`** → **软删** + 审计 |

### 2.1 权限

`get_permissions`：在父类结果上 **追加** **`IsSystemAdmin()`** — 仅 **`User.is_system_admin`** 为真可访问（见 `user/permissions.py`）。

### 2.2 `get_queryset`

- 基类已过滤 **`is_deleted=False`** 并按数据范围处理（管理员通常全量）。
- 若查询参数 **`name`** 存在：对 **`org_name`** 做 **`icontains`**。
- **`prefetch_related("members")`** 减轻列表 N+1。

### 2.3 序列化

**`OrganizationSerializer`**（`user/serialize.py`）：`Meta.fields = "__all__"`，`read_only_fields` 含审计字段；请求体可带 **`members`**、**`projects`** 的主键列表（DRF 写 M2M 的常见形态）。

## 3. 信号：组织 → 项目成员同步

**文件**：`user/signals.py`

| 信号 | sender |
|------|--------|
| `m2m_changed` | `Organization.members.through` |
| `m2m_changed` | `Organization.projects.through` |

触发动作：`action in ("post_add", "post_remove", "post_clear")` 时调用 **`_sync_org_project_members(instance)`**。

### 3.1 `_sync_org_project_members` 行为

```python
members = list(org.members.all())
if not members:
    return
for proj in org.projects.all():
    proj.members.add(*members)
```

**语义**：

- 对组织当前 **每一名成员**，向该组织 **关联的每一个项目** 执行 **`proj.members.add(*members)`**（幂等添加）。
- **若组织成员为空**：直接 `return`，**不会**从项目中移除任何用户。
- **从组织移除成员** 或 **清空成员**：当前实现 **不会** 自动 `remove` 项目成员；项目侧可能仍保留历史成员，需产品决定是否增加「反向同步」或管理员工具清理。

**从组织移除项目关联**：`post_remove`/`post_clear` 仍会进入 `_sync_org_project_members`；若此时组织仍有成员，会再次 `add` 到 **剩余** 关联项目上，仍 **不** 根据「被移除的项目」删人。

## 4. 典型前后端交互

| 动作 | HTTP | 备注 |
|------|------|------|
| 列表 | `GET /api/user/orgs/?page=1&name=关键字` | 需系统管理员 |
| 新建 | `POST /api/user/orgs/` | Body 含 `org_name`、`members`、`projects` 等 |
| 局部更新 | `PATCH /api/user/orgs/<id>/` | M2M 常以 id 列表提交 |
| 软删 | `DELETE /api/user/orgs/<id>/` | |

前端参考：系统管理 **组织管理**（如路由 `/system/org`，以 `frontend` 路由表为准）。

## 5. 显式与静默

| 显式 | 静默 |
|------|------|
| API 返回的组织、成员、项目 id | 信号触发的 `project.members.add` 无独立响应体；可能与用户期望的「移出组织即移出项目」不一致 |

---

## 6. 排障

1. **403**：当前用户非 `is_system_admin`。
2. **项目里仍有人**：属当前同步策略限制；需人工或扩展信号逻辑。
3. **列表慢**：检查是否缺少 `prefetch` 或 M2M 过大。

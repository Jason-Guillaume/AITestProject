# 敏感信息变更与待审批状态

## 文档目的

说明 **用户名 / 登录密码** 的变更为何走 **`UserChangeRequest`** 审批流；接口 **`POST /api/user/me/sensitive-change/`**、**`GET /api/user/me/change-requests-status/`**；以及 **`post_save` 信号** 如何触发 **站内信**（非邮件）。

---

## 1. 数据模型 `UserChangeRequest`

**表**：`sys_user_change_request`（`user/models.py`）

| 概念 | 说明 |
|------|------|
| `RequestType` | `username` / `password` |
| `Status` | `pending` / `approved` / `rejected` |
| `new_value` | 用户名变更存 **明文目标用户名**；密码变更存 **`make_password` 后的哈希串**（审批通过时赋给 `User.password`） |
| `user` | 申请人 FK |
| `approver` / `approved_at` | 审批人及时间 |

索引：`(status, -created_at)`。

---

## 2. 提交申请：`POST /api/user/me/sensitive-change/`

- **视图**：`UserSensitiveChangeRequestAPIView`
- **序列化器**：`UserSensitiveChangeRequestCreateSerializer`

### 2.1 请求字段

- **`request_type`**：`username` 或 `password`（choices）
- **`new_username`**：`request_type=username` 时必填且与当前不同、未被占用
- **`new_password`**：`request_type=password` 时必填明文（仅 write_only，入库前转哈希）

### 2.2 业务规则（`validate` / `create`）

- 同一用户、同一 `request_type` 已存在 **`status=pending`** 的记录 → **拒绝**：「您已有待审批的同类申请…」
- 密码分支：`new_value = make_password(validated_data["new_password"])`

### 2.3 成功响应

```json
{
  "code": 200,
  "msg": "已提交申请，请等待管理员在站内消息中审批",
  "data": { "id": <申请ID>, "status": "pending" }
}
```

---

## 3. 待审批状态查询：`GET /api/user/me/change-requests-status/`

- **视图**：`UserMyPendingSensitiveStatusAPIView`
- **返回** `data`：

```json
{
  "pending_username": true|false,
  "pending_password": true|false
}
```

用于前端禁用重复提交或展示「审核中」角标。

---

## 4. 站内信：`post_save` 信号

**文件**：`user/signals.py`

- 接收者：`UserChangeRequest`，**`created=True`** 且 **`status=pending`**。
- 使用 **`transaction.on_commit(create_inbox_messages_for_change_request)`**，避免事务未提交时管理员读不到申请。
- **`create_inbox_messages_for_change_request`**（`user/change_request_actions.py`）：对 **`admin_recipients_queryset()`**（`is_staff` OR `is_superuser` OR `is_system_admin`）中 **每一位管理员** `bulk_create` 一条 **`SystemMessage`**，并关联 **`related_request`**。

**不发送邮件**（视图 docstring 已说明）。

---

## 5. 审批执行（概要）

通过 / 拒绝逻辑在 **`approve_change_request` / `reject_change_request`**，详见 **`09`**、**`10`** 篇。通过后用户名写回 `User.username`；密码将 **`User.password = cr.new_value`**（已是哈希）。

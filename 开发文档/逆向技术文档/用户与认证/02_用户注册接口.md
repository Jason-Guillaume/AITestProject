# 用户注册接口

## 文档目的

说明 **`POST /api/user/register/`** 的请求体、校验链、用户落库方式及与 **图形验证码** 的耦合。便于安全审计（验证码一次性、密码哈希）与接口联调。

---

## 1. HTTP 路由与视图

- **路径**：`POST /api/user/register/`
- **视图**：`UserRegisterAPIView`（`user/views.py`）
- **鉴权**：`authentication_classes = []`、`permission_classes = []` — **开放注册**（若部署需关闭，应在上层网关或移除路由实现）。

---

## 2. 请求体与序列化器

**序列化器**：`UserRegisterSerializer`（`user/serialize.py`）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `username` | string | 是 | 最大长度 150 |
| `password` | string | 是 | 最大 128；由 `create_user` 做 Django 哈希 |
| `email` | email | 否 | 去空格；若填则全库唯一 |
| `phone_number` | string | 否 | 去空格；若填则全库唯一 |
| `captcha_code` | string | 是 | 用户输入的 4 位图形码（write_only） |
| `captcha_uuid` | string | 是 | 与 `GET /api/user/captcha/` 返回的 **`uuid`** 对应 |

**注意**：序列化器 **未** 将 `real_name` 暴露为输入字段；`create` 中 `real_name` 使用 **`validated_data.get("real_name", "")`**，默认空字符串。

### 2.1 `validate` 逻辑摘要

1. 从缓存读 `captcha_{uuid}`；不存在 → `ValidationError`「验证码已过期或无效」。
2. 与 `captcha_code` **忽略大小写**比较；失败 → `ValidationError`「验证码错误」。
3. **成功后 `cache.delete`** 该 uuid 键（防重放）。
4. 弹出 `captcha_code`、`captcha_uuid`，不写入用户表。
5. `email` / `phone_number` 非空时查重：`User.objects.filter(...).exists()`。

### 2.2 `create` 逻辑

使用 **`User.objects.create_user(...)`**（**禁止** `User.objects.create` + 明文密码，注释已强调）。参数：`username`、`password`、`real_name`、`email`、`phone_number`。

---

## 3. 成功响应

```json
{ "code": 200, "msg": "注册成功", "data": null }
```

失败时由 DRF **`serializer.is_valid(raise_exception=True)`** 抛出，一般为 **400**，`detail` 或字段错误字典。

---

## 4. 数据库

- **表**：`sys_user`（模型 `user.models.User`，继承 `AbstractUser` + `BaseModel`）
- **注册成功**不写审计事件（本接口未调用 `record_audit_event`）；若需合规可后续在 `create` 中扩展。

---

## 5. 前端对接

- **`frontend/src/api/auth.js`**：`registerApi(payload)` → `POST /api/user/register/`。
- 须先调 **`captchaApi`** 取得 `uuid` 与 `image`，表单提交 **`captcha_uuid`**（映射 uuid）与 **`captcha_code`**。

---

## 6. 显式与静默

| 显式 | 静默 |
|------|------|
| 成功仅 `msg` | 密码不以明文出现在任何成功响应中 |
| 校验错误中的字段名 | Django `AbstractUser` 内置字段的其它约束由 ORM/模型层触发 |

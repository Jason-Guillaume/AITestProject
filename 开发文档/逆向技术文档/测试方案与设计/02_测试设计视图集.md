## 2. 回收站模式 `get_queryset`

辅助函数：**`_recycle_mode_from_params(params)`**（同文件上文），兼容：

- **`recycle=1` / `true` / `yes`**
- **`is_deleted=true` 等等价真值字符串**

回收站分支逻辑摘要：

```text
qs = TestDesign.objects.filter(is_deleted=True)
若非管理员且启用数据范围 → _apply_member_scope(qs, user)
排序：order_by("-update_time", "-id")
```

非回收站：**`return super().get_queryset()`** —— 与普通 CRUD 一致，仅 **`is_deleted=False`** 加范围。

前端切换「垃圾桶」时需 **整条列表重新请求**，不可仅在前端筛选，否则分页总数与后端不一致。

## 3. `POST .../designs/batch-delete/`

**Body**：**`{ "ids": number[] }`**

解析器：**`_parse_id_list`**：

- **`ids`** 必填且为数组；元素逐项 **`int`** 化，非法跳过，**≤0** 丢弃；
- 去重保持稳定顺序；

**查找**：在当前 **`get_queryset()`** 结果上 **`filter(id__in=id_list, is_deleted=False)`** —— **回收站模式下仅操作已删行时可能找不到目标**，一般以未删列表调用。

删除循环：

- **逐条 **`with transaction.atomic(): self.perform_destroy(obj)`**；
- **动机**（源码注释）：单条 **`perform_destroy` 抛出异常会令外层 `atomic` 块进入 PostgreSQL/MySQL 的「失败状态」，后续语句不可继续**；此处用「每迭代独立 atomic」等效 savepoint，使 **部分成功 + errors 明细** 可用。

响应字段：

- **`success`**：无任一错误时为 true；
- **`deleted`**：成功条数；
- **`missing_ids`**：请求中存在但 queryset 未命中的 id；
- **`errors`**：**`[{ "id", "error" }]`**。

## 4. `POST .../designs/batch-update/`

**Body**：**`{ "ids": [], "patch": {} }`**

校验：

1. **`request.data` 须为 dict**；
2. **`patch` 为非空 dict**；
3. **`allowed_keys`**（源码集合）：  
   **`design_name`、`req_count`、`point_count`、`case_count`、`review_status`、`archive_status`**  
   其它键静默丢弃；
4. 若 **`cleaned` 为空** → **400**「patch 未包含可更新字段」。

查询 **未删除** 且 **`id__in`** 的记录后 **逐条 setattr + save**，错误收集策略与 batch-delete 类似（读源码时注意是否同样使用 isolated atomic）。

## 6. 与模块、用例的业务关系

**`TestDesign`** 常与 **测试方案 `TestApproach`、模块 `TestModule`、项目** 编排同一「规划流水线」：**`design_name`/`case_count`** 与看板 KPI 对齐。删除设计 **不等价** 于删除底下用例，除非另有级联或服务层脚本。

## 7. 前端联调清单

| 场景 | 建议 |
|------|------|
| 批量改状态 | 使用 **`batch-update`**，勿对几十条单行 PATCH |
| 回收站恢复 | 使用标准 **`partial_update`** 将 **`is_deleted`** 复位（若序列化允许）或专用 restore 动作（若项目后续新增） |
| 错误反馈 | 展示 **`errors` 数组** 中单条失败原因 |

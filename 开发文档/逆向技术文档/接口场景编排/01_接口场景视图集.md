# 接口场景视图集

## 文档目的与读者

说明 **`ApiScenario`** 的资源模型、**`/api/execution/api-scenarios/`** 标准 CRUD、**`BaseModelViewSet`** 下的 **项目成员数据范围**，以及 **`POST .../<id>/run/`** 如何创建 **`ApiScenarioRun`** 并投递 Celery。适合对接场景编排页、排查「run 无反应」或权限导致的列表为空。

---

## 1. 路由与类

| 项 | 值 |
|----|-----|
| 前缀 | **`/api/execution/`**（见 `AITestProduct/urls.py` → `include("execution.urls")`） |
| Router | `execution/urls.py`：`router.register(r"api-scenarios", ApiScenarioViewSet)` |
| 类 | `ApiScenarioViewSet`（`execution/views.py`） |
| Serializer | `ApiScenarioSerializer`（`execution/serialize.py`） |

---

## 2. 数据范围与软删

- **`ApiScenario.project`** 为 FK → **`TestProject`**，`BaseModelViewSet._apply_member_scope` 命中 **`project__members` 或 creator**。
- 列表默认 **`is_deleted=False`**；无单独回收站扩展时与其它 execution 资源一致（以实际 ViewSet 是否覆盖 `get_queryset` 为准，当前 **`ApiScenarioViewSet` 未覆盖**）。

---

## 3. 标准 REST

| 操作 | 说明 |
|------|------|
| GET list | 支持 DRF 分页（若全局配置） |
| POST create | Body 含 `project`、`name`、`environment`（可空）、`default_variables`、`failure_strategy`、`is_active` 等，与模型及 Serializer 一致 |
| GET/PATCH/DELETE detail | 软删走基类 `perform_destroy` |

**Serializer**：`fields = "__all__"`，`read_only_fields` 含 `creator/updater/create_time/update_time`，附带 **`creator_name`**。

## 4. 自定义动作：`run_scenario`

**路径**：`POST /api/execution/api-scenarios/<pk>/run/`  
**`url_path`**：`run`（`detail=True`）。

### 4.1 前置校验

| 条件 | 结果 |
|------|------|
| **`scenario.is_active`** 为假 | **`ValidationError`**：`{"msg": "场景未启用", "code": 400, "data": None}` |
| Body 中 **`variables`** 存在且 **非 dict** | **`ValidationError`**：`variables 必须为对象` |

### 4.2 环境 ID

- 从 Body 读 **`environment_id`**；若 **`None`** 则回落 **`scenario.environment_id`**。
- 若 Body 提供了非整数，**`int()` 失败会将 `env_id` 置为 `None`**（即忽略非法值，再走场景默认环境）。

### 4.3 变量合并顺序

1. **`scenario.default_variables`**（dict，缺省 `{}`）  
2. **`request.data["variables"]`**（若为 dict）**update 覆盖**

合并结果写入 **`ApiScenarioRun.initial_variables`**，并作为 Celery **`overrides["variables"]`** 传入。

### 4.4 创建运行记录与异步任务

1. **`ApiScenarioRun`**：`status=pending`，`environment_id` 快照，`initial_variables`/`final_variables`/`summary` 初值，`creator/updater` 为当前用户（若已认证）。
2. **`run_api_scenario_run.delay(run.id, overrides={"variables": init_vars})`**
3. **`ApiScenarioRun.objects.filter(pk=run.pk).update(celery_task_id=...)`**
4. **`record_execute_audit`**：`extra` 含 **`scenario_id`、`scenario_run_id`**

### 4.5 响应

```json
{
  "success": true,
  "run_id": 123,
  "celery_task_id": "<celery-async-result-id>"
}
```

## 5. 与步骤、运行的关系

| 实体 | 关系 |
|------|------|
| **`ApiScenarioStep`** | `scenario` 外键；每步挂 **`TestCase`**（多为 API 子类型） |
| **`ApiScenarioRun`** | 一次「跑批」；由 **`run_scenario`** 或未来其它入口创建 |
| **`ApiScenarioStepRun`** | 每步执行结果；由 **`scenario_runner.run_api_scenario`** 写入 |

详见《03》《05》《06》。

---

## 6. 排障

| 现象 | 检查点 |
|------|--------|
| 列表无场景 | 当前用户是否项目成员；场景是否软删 |
| run 返回 400 未启用 | `is_active` |
| 有 `run_id` 但状态不变 | Celery worker、`run_api_scenario_run` 日志；**重复执行**时若 run 已是 **`running`/`success`**，**`run_api_scenario` 会直接跳过**（见《05》） |
| 变量未生效 | 确认 **`variables`** 为 JSON 对象；与 **`default_variables`** 合并顺序 |

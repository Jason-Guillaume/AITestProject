# 流水线日志写入与查询链路

## 文档目的

把 **日志从子进程字节流写入 `pipeline_log` 表的完整链路**拆成：**线程读到队列、`Queue` drain、单行 `PipelineLog`、`Pipeline.status` 终态**四段；并解释 **增量 HTTP 日志接口**如何用 **主键 `id` 游标**与时间戳排序协同，服务于前端轮询与排障 **`Running` 卡死**。源码真值：**`project/tasks.py`、`project/views.PipelineViewSet.logs`**。

---

## 1. 写入端：`run_pipeline_task`

### 1.1 任务入口绑定

|**项**|**说明**|
|-----|-------|
|**装饰**|**`@app.task(bind=True, name=\"run_pipeline_task\")`**|
|**导入路径**|`**`from project.tasks import run_pipeline_task`**（ViewSet **`run`**）|
|**幂等**|**多次 `delay` → 并行多进程**——若需独占锁应在外层 Pipeline 维度加 **`select_for_update`**（当前实现未加）|

任务开始：

1. **`Pipeline.objects.get(id=pipeline_id)`**；
2. **`pipeline.status = RUNNING`、`save`** ——控制台应立即 **刷新状态条**；

若记录不存在：**静默 return**——客户端仍收到 **`task_id`** 却无 DB 演化，须在 **Flower** **查丢弃原因**。

### 1.2 子进程

**`command` 默认值**：`**`echo \"No command supplied for pipeline\"`** ——调试 **无害输出**；

**`Popen`** 参数：**`shell=True`、`stdout/err=PIPE`、`text=True`、`bufsize=1`（行缓冲）**；

**安全风险**：整条字符串进 shell ——参阅 **`01`** 文档 **§7**。

### 1.3 .reader 线程模型

|**函数**|**职责**|
|--------|-------|
|**`_pipe_to_queue(pipe, prefix, out_q)`**|循环 **`readline`**，非空：`out_q.put((prefix, stripped_line))`; **prefix**：**`stdout`/`stderr`**|

两线程：**stdout**、**stderr** 并行，`Queue` FIFO 粗略融合时间序；**单行内顺序保真，跨通道交错可能**。

结束时 **`pipe.close()`** `finally`。

### 1.4 落库 drain

|**函数**|**语义**|
|--------|-------|
|**`_drain_queue_to_db(pipeline, out_q)`**|**`get_nowait` 暴风读** → **`PipelineLog.objects.create`**，文本 **`f\"[{prefix}] {line}\"`**|

**粒度**：一文一行 DB insert ——吞吐 **≤百行/s** OK；>**1k lines/s** DB 可能成为瓶颈。**优化方向**：**`bulk_create` 分批**——当前 **未启用**。

主循环：**`sleep(0.05)`**，条件：**进程未结束 ∪ 读到线程存活 ∪ queue 未空**。退出后：**`proc.wait()`**、`drain` 再次。

## 2. 终态与异常

|**路径**|**status**|**日志**|
|--------|----------|-------|
|**`return_code == 0`**|`**`STATUS_SUCCESS`(2)**|可能无stderr|
|**非零**|**`STATUS_FAIL`(3)**|stderr 常含报错|
|**Exception**|**`FAIL`**|单行 **`PipelineLog` 写入 `str(exc)`**|

最后 **`finally`**：若进程仍存在 **`terminate()` → wait → kill`**，防 **僵尸子 shell**。

---

## 3. 查询端：`PipelineViewSet.logs` 对齐

REST：**`GET /api/project/pipelines/<pk>/logs/?after_id=0&limit=500`**。

ORM 草稿：

```python
qs = PipelineLog.objects.filter(pipeline=pipeline, id__gt=after_pk).order_by("id", "timestamp")[:limit]
```

|**设计点**|**动机**|
|------------|-------|
|**主键游标**|**`timestamp`相同**时仍可 **单调推进**|
|**双层 order**|与时间审计轨迹一致|
|**上限2000**|**防单次拉爆 payload**|

视图先 **`refresh_from_db(fields=[\"status\"])`**：**状态与增量日志同一响应**可减少竞态错觉。

|**响应字段**|**说明**|
|--------------|-------|
|**`pipeline_status`**|整型 **`0～3`**|
|**`items`**|serializer 字典列表|
|**`last_log_id`**|客户端 **下一轮 `after_id`**|

若 **`items=[]` 且仍 RUNNING**：**可能仍在静默执行**——继续轮询；若 **FAIL/SUCCESS**：**最后仍空**：**正常结束或无输出**。

---

## 4. 与 Celery **`task_id`**

HTTP **`run`** 返回 **`task_id`** **仅用于**：运维对账 **或** Celery **撤销**，**不参与 logs** 路由；控制台 **只靠 `pipeline_id`**。

混淆排查：**以为 task 失败等价 pipeline fail**——**错误：task 外层 try/except 已写 **`PipelineLog` + FAIL**。

---

## 5. 典型故障矩阵

|**现象**|**可能根因**|**核对**|
|----------|----------------|-------|
|**无日志 yet RUNNING**|**命令不产生 stdout/stderr、缓冲满**|往 command 前缀 **`PYTHONUNBUFFERED=1`** 等|
|**巨量重复行**|`**`poll`结束前 drain 不彻底**|检查 **`exit`后二次 drain**已实现|
|**404 logs**|`**`get_queryset`** 剔除软删 **`Pipeline`**|确认对象 **未软删 scope**|
|**status 卡住 RUNNING**|**Worker SIGKILL、`save`未成**|Redis broker、DB连接；**Flower** traceback|

---

## 6. DB 体量与归档

|`pipeline_log` 单行 **TextField**——**超长 shell 输出 → 磁盘**膨胀；运维应 **cron 截断**：**`PIPELINE_RETENTION_DAYS`**（若未实现请加 story）。

归档策略：**移动到冷存储 / S3**，主表仅存 **偏移指针**——需 **离线工具**协调。

---

## 7. 前端轮询节流

建议使用：**`minimum interval 250ms`** + **末尾加速到 500ms**：平衡 **时效 vs QPS**。网络错误 **指数退避**，防 **Thundering herd**。

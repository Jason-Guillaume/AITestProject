# 用例版本与逻辑删除

## 文档目的

从 **ORM Manager、编号唯一约束、软删恢复、物理删除与序号重排、版本快照与 rollback** 五条主线，逆向说明 **`testcase/models.py`** 中 **`TestCase` / `TestCaseVersion`** 与 **`testcase/views.py`** 上回收站动作的对应关系；面向排障 **`case_number` 冲突、`rollback-version` 后步骤错位**。

---

## 1. 软删字段语义

与全局 **`common.models.BaseModel`** 对齐：

- **`is_deleted`**：逻辑删除标记；列表默认 queryset 不包含已删。
- **`deleted_at`**：**`TestCase`** 侧用于回收站时间展示（若非所有模型皆有，以 **`TestCase` 字段定义** 为准）。

**`perform_destroy`**：在 **`BaseModelViewSet`** 与用户自定义 **`TestCaseViewSet.perform_destroy`** 链路中通常为 **软删**，不触发 SQL **`DELETE`**；与 **项目的 `hard_delete` 视图**对照阅读。

---

## 2. 三个 Manager（概念表）

定义于 **`testcase/models.py`** 的 **`TestCase`**：

| Manager | 典型过滤 |
|---------|----------|
| **`objects`** | `is_deleted=False` —— 正常运行时 CRUD |
| **`all_objects`** | 无条件 —— **`Max(case_number)`、对比历史** |
| **`deleted_objects`** | `is_deleted=True` —— **`restore`、`hard-delete` 视图** |

**`restore` / `hard-delete` 视图**：**`get_object_or_404(TestCase.deleted_objects, pk=pk)`**，避免误恢复未删实例。

列表「回收站」模式需 **显式 queryset**（见 **`01` 篇** **`get_queryset`**）；不可混用默认 **`objects`**。

---

## 3. `case_number` 与 `test_type`

- **Partial unique**：**`(test_type, case_number)`** 仅在 **`is_deleted=False`** 时唯一（PostgreSQL **`UniqueConstraint` with condition** / MySQL 用 **`is_deleted` 参与 uniq** 的常见折中）。
- **分配策略**：模型 **`save()`** 时在事务内 **`select_for_update`** 扫描 **`TestCase.all_objects`** 同类型的 **`Max(case_number)`**，新号 = max+1。  
  - 动机：回收站占位 **不阻断** 新编号增长；硬删后可 **紧凑型重排**（见§4）。
- **`type_sequence`** 若与 **`case_number`** 同义请以代码注释为准。

---

## 4. `hard_delete` 与编号重排

**`TestCase.hard_delete`**（实现细节以模型为准）典型步骤：

1. **事务.atomic**；
2. 记录 **`removed_no = case_number`、`tt = test_type`**；
3. **`super().delete()`** —— **真实物理删除** 主行及 DB 层 **级联的子表 OneToOne**（若 **`on_delete=CASCADE`**）；
4. 对 **`test_type=tt` 且 `case_number > removed_no` 且未删** 行批量 **`F("case_number") - 1`**，保持连续编号。

**风险**：若存在 **外链引用 execution 日志**仅保存 **`case_number` 而无 id**，物理删会破坏历史可追溯性 —— **产品上以软删为准**，硬删仅回收站慎用。

---

## 5. 版本：`TestCaseVersion` 与快照

版本行保存：

- **`case_snapshot`**：JSON blob，内含主表字段片段、各 **`test_type` 子表段**、`steps` 数组等；
- **`release_plan`、`creator`** 等在列表接口 **`select_related`** 预取；

**`GET .../cases/<id>/versions/`**：分页列出，按创建时间逆序常见。

---

## 6. `POST .../cases/<id>/rollback-version/` 行为摘要

请求体：**`{"version_id": <int>}`**。

顺序（概念）：

1. 载入目标 **`TestCaseVersion`**，校验 **`case`** 归属；
2. **`TestCaseVersion.create_version`**：**先写入「回溯前当前状态」为新版本** —— **基线可追溯**；
3. **`case.apply_snapshot_payload(snapshot)`**：  
   - 更新主字段；  
   - 对各子类型块调用 **`_apply_subtype_snapshot`**；  
   - 步骤：**软删旧 **`TestCaseStep`** 再按快照重建**，避免幽灵步骤 id。

成功后响应常含 **`rollback_to_version_id`、`baseline_version_id`**（见源码字符串键名）。

---

## 7. 其它相关 `@action`

- **`batch-execute`**：**`exec_count`** 聚合 +1 —— **非真实 HTTP 执行**，勿与 **`execute-api` 混淆**。
- **`execute-api` / `run-api`**：**真正跑接口**，写 **`ExecutionLog`**；差异在是否写 **`ApiTestLog`** 遗留表（参阅 **`07` 篇**）。

## 8. 排障备忘录

| 现象 | 检查 |
|------|------|
| 创建报唯一约束冲突 | 是否有 **同类型未删** 占号；是否在并发下缺 **`select_for_update`**（迁移/DB 隔离级别）。 |
| 回滚后 UI 仍是旧断言 | **`apply_snapshot_payload` 是否失败** silently；步骤是否缓存；前端是否未 **invalidate**。 |
| 回收站为空 | queryset 是否正确切到 **`all_objects`/deleted**。 |

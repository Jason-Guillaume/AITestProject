# 06 — 用例服务层与仓储（逆向）

## 1. API 执行核心 `testcase.services.api_execution`

- `run_api_case`：执行单条 API 用例的主入口（装配 headers/body/变量替换后发 HTTP，写 `ExecutionLog`）。
- `preview_resolved_request`：执行前预览真实 URL/头/体（`views.ApiScenario` 或单用例调试场景会用到）。

## 2. 子类型与配置 `testcase.services.case_subtypes`

- `create_typed_case(test_type, base_fields, api_payload, ...)`：统一创建功能/接口/性能/安全/UI 等子表记录。
- `get_api_profile_for_execute`：取出 API 用例执行所需 profile（URL、方法、断言配置等）。

## 3. AI 相关服务

- `testcase.services.ai_openai`：`ai_fill_test_data`、`ai_import_api_cases`（Swagger/cURL → LLM JSON 数组）。
- `testcase.services.ai_case_gate`：`gate_ai_api_cases` —— 在 `ApiImportFromSpecAPIView` 创建用例前对 LLM 输出做规则校验/归一。
- `testcase.services.ai_import_precheck_core`：`precheck_api_draft_item` —— `TestCaseViewSet.ai-import` 路径的预检。

## 4. 变量与抽取 `testcase.services.variable_runtime`

- `suggest_extractions`：绑定至 `POST /api/testcase/suggest-extractions/`，从响应 JSON 推断可变量化路径。

## 5. 视图与服务的边界

- `TestCaseViewSet` 体积较大：分页/筛选/子类型序列化在视图；**执行/AI/导入** 下沉到 services，便于单测与复用。

## 6. 扩展指南

- 新增执行引擎（如 gRPC）应仿照 `api_execution` 的分层，不要在 view 里直接 `requests.request`。

## 附录 B：请求 / 响应形态（逆向归纳）

1. **认证**：除明文说明外，REST 接口默认需要登录态（常见为 Token / Session，以项目中间件为准）。
2. **分页**：列表接口若使用 `PageNumberPagination`，关注 `page`、`page_size` 上限与默认排序字段。
3. **幂等**：创建类 POST 与异步 202 类接口需在前端防重复点击；工单 / 导入类注意业务幂等键（如 `version_no`、用例名）。
4. **软失败**：部分模块刻意返回 HTTP 200 + `success: false`，避免全局 Axios 拦截器误判；集成时以响应体为准。

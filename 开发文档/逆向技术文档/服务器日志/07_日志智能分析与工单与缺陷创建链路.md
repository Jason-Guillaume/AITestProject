# 07 — 日志智能分析、工单草稿与缺陷链路（总览逆向）

## 1. 能力分层

| 能力 | 入口 | 同步性 | 落库 |
|------|------|--------|------|
| 单行/纯文本 AI 诊断 | `POST .../analyze/` | 同步 | 仅审计 |
| ES 增强诊断 | `POST .../analyze-with-context/` | 同步 | 审计 + ES meta |
| 异步工单草稿 | `POST .../auto-tickets/enqueue/` | 异步（Celery） | `LogAutoTicketJob` |
| 工单结果查询 | `GET .../auto-tickets/jobs/<id>/` | 同步轮询 | 读 job |
| 从草稿创建缺陷 | `POST .../create-defect/` | 同步 | `TestDefect` + job 关联 |

## 2. AI 模块边界

- 诊断 Markdown：`server_logs.ai_analyze`（单条与带上下文函数分离）。
- 工单结构化草案：`server_logs.ai_ticket.analyze_log_ticket_draft`（由 Celery 任务调用）。
- 缺陷实体：`server_logs.defect_create.create_test_defect_from_auto_ticket`（REST 手动或任务内自动）。

## 3. Celery 任务 `run_log_auto_ticket_job`

- 乐观锁：`PENDING` → `PROCESSING`，避免重复执行。
- 上下文：`fetch_es_context_for_anchor` 与同步 API 共用路径，保证「前台看到的锚点」与「后台工单」一致。
- 失败：`FAILED` + `error_message`；成功：`SUCCESS` + `draft` JSON；可选自动创建缺陷时写 `created_defect` 并记审计。

## 4. 审计与执行审计

- 模块内统一 `ServerLogAuditEvent`（analyze/search/auto_ticket/host_* 等）。
- 入队额外 `record_execute_audit`，便于在通用审计台帐检索「谁在什么时间触发了自动化工单」。

## 5. 限流与配额

- `AnalyzeLogAPIView`、`AnalyzeLogWithContextAPIView`、`LogAutoTicketEnqueueAPIView`、`LogAutoTicketCreateDefectAPIView` 使用 `ScopedRateThrottle` scope `server_logs_analyze`，防止模型费用与连接被刷爆。

## 6. 产品级注意

- 上游模型错误在单条 analyze 路径刻意 HTTP 200；异步工单路径则以 job 状态表达失败。
- 凭据从不写入 `LogAutoTicketJob`，仅任务 kwargs 传递，符合最小敏感面原则。

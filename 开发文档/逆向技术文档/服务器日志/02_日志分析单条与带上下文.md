# 02 — 日志分析：单条与带 ES 上下文（逆向）

## 1. 单条分析 `AnalyzeLogAPIView`

- 路由：`POST /api/server-logs/analyze/`，对应 `server_logs.urls` 中 `AnalyzeLogAPIView`。
- 权限：`IsAuthenticated`。
- 限流：`ScopedRateThrottle`，`throttle_scope = "server_logs_analyze"`（与其它分析类接口共享配额维度，具体限频在 DRF settings 配置）。
- 请求体：经 `LogAnalyzeRequestSerializer` 校验，核心字段为 `log_text`；可选 `model`、`api_key`、`api_base_url` 用于覆盖默认模型调用凭据。
- 行为：调用 `server_logs.ai_analyze.analyze_log_markdown(...)`，返回 Markdown 诊断。
- 错误处理：**故意返回 HTTP 200 + `success: False`**，使前端统一拦截器不把「上游模型失败」当成 5xx；同时写 `ServerLogAuditEvent.Action.ANALYZE`，`meta` 中带 `success` 与截断后的 `error`。

## 2. 带上下文分析 `AnalyzeLogWithContextAPIView`

- 路由：`POST /api/server-logs/analyze-with-context/`。
- 设计意图（见视图 docstring）：用 `server_id` + `anchor_text`（可选 `anchor_ts`）在 Elasticsearch 中定位锚点，再取前后 `window_seconds` 窗口内的日志行，与锚点一并送入模型，减轻「断章取义」。ES 不可用或查询失败时可退化为仅分析 `anchor_text`（由 `fetch_es_context_for_anchor` 与 `analyze_log_markdown_with_context` 协作完成）。
- 权限：先校验用户是否对 `server_id` 对应主机有权（`remote_log_server_queryset_for_user`）；无权返回 **403** + 统一 JSON 形态。
- `meta`：`context_count`、`backend`、`es_error`（若有）会进入审计。

## 3. 与 ES 索引的关系

- 历史上下文依赖写入 ES 的文档结构（至少 `server_id`、`message`、`timestamp` 等）。实时 tail 在 `server_logs/ssh_tail.py` 中异步入队/bulk 索引；若 ES 未部署，`analyze-with-context` 仍可调用模型，但上下文行数可能为 0。

## 4. 合规与运维

- 审计不落敏感凭据；分析请求可在 `meta` 中记录字符数、模型名、成功与否。
- 模型 API Key 若由前端传入，应注意 HTTPS 与日志脱敏（代码层仅传递至 AI 模块，不写入 `LogAutoTicketJob` 等表）。

## 5. 前端对接提示

- 单条分析失败看响应体 `success`/`error`，勿仅用状态码。
- 带上下文接口建议传 `anchor_ts`（毫秒）以提高锚点唯一性。

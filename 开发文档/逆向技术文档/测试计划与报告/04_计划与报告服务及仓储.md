# 计划与报告服务及仓储

## 文档目的与读者

说明 **`TestPlanService` / `TestReportService`** 与 **`TestPlanRepository` / `TestReportRepository`** 的职责边界、校验项、以及与 **ViewSet 直写** 的差异。适合新增「从消息队列创建计划」等后台入口时复用同一业务规则。

---

## 1. 分层位置

| 层 | 路径（节选） |
|----|----------------|
| Service | `execution/services/test_plan_service.py`、`execution/services/test_report_service.py` |
| Repository | `execution/repositories/*`（由 Service `__init__` 注入） |
| ViewSet | `execution/views.py` — `TestPlanViewSet`、`TestReportViewSet` |

二者 **共用同一套 ORM 模型**；Service 更强调 **显式校验 + 事务 + 业务异常类型**（`ValidationException`、`DuplicateException`、`ResourceNotFoundException`）。

---

## 2. `TestPlanService` 要点

### 2.1 `create_test_plan(data, creator)`

- **必填**：`plan_name`、`version_id`、`environment`（字符串非空）。
- **名称唯一**：`repository.exists_by_name`。
- **创建字段**：含 `plan_status`（默认 **1**）、`iteration`（默认 `''`）、日期可选。
- **测试人员**：若 `data['tester_ids']` 存在且非空，`User.objects.filter(id__in=...)` 后 **`test_plan.testers.set`**。

### 2.2 `update_test_plan(plan_id, data, updater)`

- 存在性检查；若改 **`plan_name`** 再跑唯一校验（排除自身 id）。
- 白名单字段批量写入：`plan_name`、`version_id`、`iteration`、`environment`、`plan_status`、`start_date`、`end_date`。
- **`tester_ids`**：若提供则整体 **`set`**。

> ViewSet 的 **`batch-update`** 则走 **Serializer 白名单**，与 Service 的字段集合 **不必完全一致**；后台任务应择一为主，避免两套规则漂移。

---

## 3. `TestReportService` 要点

### 3.1 `create_test_report(data, creator)`

- **必填**：`report_name`、`plan_id`、`project_id`。
- **名称唯一**：`repository.exists_by_name`。
- **默认**：`create_method` 默认 **1**；`trace_id` 默认 `''`；`start_time`/`end_time` 默认 **`datetime.now()`**（若调用方未传）。
- 初值统计：`case_count`、`pass_rate`、`defect_count` 等可由调用方传入或默认 0。

### 3.2 `update_test_report` / `update_statistics`

- `update_test_report`：允许更新子集字段（`report_name`、`environment`、`case_count`、`pass_rate`、`defect_count` 等），名称唯一同样生效。
- `update_statistics`：见文件后续方法（供报告跑完后写回聚合指标）。

---

## 4. Repository 模式

Repository 封装 **`get_by_id`、`create`、`update`、`exists_by_name`** 等，便于单元测试 **mock 数据层**。ViewSet **默认不强制** 经过 Service；若希望 **单一写入路径**，可渐进式把 `perform_create` 代理到 Service（当前仓库以实际代码为准）。

---

## 5. 与 Celery 任务的关系

**`run_test_report_execute`** 直接操作 **`TestReport` / `TestCase` / `ReleasePlanTestCase`** ORM，并更新报告统计与追溯字段；**不一定**调用 `TestReportService.update_statistics`（以 `execution/tasks.py` 实现为准）。  
排障时应 **同时打开 tasks 与 service**，避免「只改 Service 却无效」。

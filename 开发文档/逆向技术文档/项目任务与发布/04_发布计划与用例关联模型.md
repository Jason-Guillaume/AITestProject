# 发布计划与用例关联模型

## 文档目的

从 **ORM 约束、M2M-through 软删、聚合服务 `build_release_risk_brief`** 三个层次，固定 **`ReleasePlan` / `ReleasePlanTestCase`** 与 **`testcase.TestCase`** 的逆向契约；配合 **`ReleasePlanViewSet`**（**`03`** 篇）接口时，可正确理解 **「发布 id」** 在 **`execution.TestPlan.version_id`、`defect.TestDefect.release_version_id`** 中的一致命名。

---

## 1. `ReleasePlan`（表 `release_plan`）

**定义**：**`project/models.py`**。

| 字段（节选） | 说明 |
|--------------|------|
| `project` | **FK → `TestProject`**，**`null=True`**；**`related_name="release_plans"`** |
| `release_name` / `version_no` / `release_date` | 业务展示与排期 |
| `status` | **1 待发布 / 2 已发布 / 3 已取消**（整型 choices） |
| `test_cases` | **M2M → `testcase.TestCase`**，**`through="ReleasePlanTestCase"`**，**`blank=True`** |

**`Meta.constraints`**：**`UniqueConstraint(fields=["project","version_no","is_deleted"], ...)`**  

利用 **`is_deleted`** 参与唯一性，在 **MySQL 无部分唯一索引** 的常见部署下，仍能表达 **「同一项目下同一 `version_no` 仅有一条未删记录」**。

**`BaseModel`** 继承：**`is_deleted`、审计、时间戳** 等与 **`TestProject`** 一致。

---

## 2. `ReleasePlanTestCase`（中间表 `release_plan_test_case`）

显式 **Through 模型**，字段要点：

| 字段 | 语义 |
|------|------|
| `release_plan` / `test_case` | **双 FK**，级联删除随主表策略 |
| `create_time` / `update_time` | 自动化时间 |
| `is_deleted` | **解除关联可走软删**，保留审计痕迹 |

**唯一约束**：**`(release_plan, test_case, is_deleted)`**，与上节同理实现 **「未删关联唯一」**。

**不写物理删**时可 **重复挂接**——先 **`is_deleted=True`** 占位，再以 **`is_deleted=False` 新开一行**（依产品迁移策略）。

---

## 3. Django M2M 与 QuerySet 注意

默认 **`release.test_cases`** 管理器是否 **过滤中间表 **`is_deleted=False`**——以 **`RelatedManager`/自定义 through** 为准；**`build_release_risk_brief`** 显式使用：

```python
ReleasePlanTestCase.objects.filter(release_plan_id=rid, is_deleted=False)
```

故 **简报与「发布计划视图」若出现数量不一致**，应检查 **serializer 暴露的计数是否走了不同路径**。

---

## 4. `build_release_risk_brief`（`project/services/release_risk_brief.py`）

**入参**：**`release: ReleasePlan`**，**`execution_days` 钳位 1～90**。**`since = now - timedelta(days)`。

**数据来源（节选）**：

1. **`ReleasePlanTestCase`**：**`linked_case_count`**、**`case_ids` 列表**；
2. **`TestPlan.objects.filter(version_id=rid)`**：**`version_id`** 在此处 **等价发布计划主键**，与 **`TestPlan` 建模命名**对齐；
3. **`TestDefect.objects.filter(release_version_id=rid)`**：计数分桶 **`by_severity` / `by_status`**；
4. **`ExecutionLog`**：窗口内 **`test_case_id__in(case_ids)`** 是否产生记录 → **`never_executed_case_ids`**（见 **`_never_executed_case_ids_in_window`**）。

**无 LLM**：纯 ORM 聚合，可作 **离线报表单测.fixture**。

## 5. 与 **`ReleasePlanViewSet`** 动作的衔接

|**HTTP**|**内部调用**|
|--------|-----------|
|**`GET .../risk-brief/`**|`**`build_release_risk_brief(release, execution_days=days)`**|
|**`GET .../never-executed-cases/`**|简报取 **`never_executed_case_ids`** → **`TestCase.only(...)`** 拼模块名|
|**`export.csv`（未执行导出）**|同上 + **StreamingHttpResponse** + **`record_export_audit`**|

**`never-executed-cases`** 的 **`limit` 上限 5000**；导出接口 **上限 200000**（请读 **`views.py`** 防 DoS）。

## 6. 创建权限：`perform_create`

非管理员：**必须是 **`project.members`**** 中的一员，否则 **`ValidationError({\"project\": ...})`**。管理员可代建。**`serializer.save(creator=user)`** 填充创建人。

## 7. 前端映射

Vue 常以 **`release.id`** 拉 **`risk-brief`** 卡片：**`plans_total`、`defects_open`、`never_executed_case_ids.length`**。

**`axios`**：**`/api/project/releases/:id/`** 与简报分离请求，利于 **SSR/骨架屏并行**。

## 8. 迁移与数据修复

|**场景**|**建议**|
|--------|--------|
| **`version_no` 冲突**|检查 **uniq 三元组**：软删后再建同名新版 |
| **中间表幽灵行**|cron **物理清理 **`is_deleted=True` 超过 N 天的行**|
| **简报用例数为 0**|计划未 **`ReleasePlanTestCase` 挂载**而是用 **serializer 写隐性 M2M**（若存在双路径则需统一）|

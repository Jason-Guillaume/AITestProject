# 03-核心业务逻辑与模块说明

## 0. 需求来源与开发动因

- 业务价值摘要：打通业务链路认知，提升需求实现与回归验证的一致性。
- 业务背景：业务模块持续扩展，需要对“模块能力、对象关系、核心流程”形成统一说明。
- 现状痛点：仅依赖代码难以快速理解业务闭环，跨模块联动场景容易产生认知断层。
- 建设目标：从业务视角梳理核心逻辑与关键链路，形成统一事实基线。
- 预期收益：提升需求评审、开发实施与回归测试的一致性与沟通效率。

## 2. 用户与组织模块 (`user/`)

### 2.1 核心模型

#### User 模型（系统用户）
```python
- username: 登录用户名
- password: Django 哈希密码
- real_name: 真实姓名
- phone_number: 手机号
- avatar: 头像（上传到 user_avatars/）
- is_active: 账号状态（正常/禁用）
- is_system_admin: 系统管理员标记
```

**关键逻辑**：
- 继承 `AbstractUser` + `BaseModel`（软删除、创建/更新时间）
- 自定义 `groups` 和 `user_permissions` 关系避免与 Django 默认冲突
- 软删除通过 `is_deleted` 字段控制

#### UserChangeRequest（敏感信息变更申请）
```python
- user: 申请人
- request_type: "username" / "password"
- new_value: 明文用户名 或 Django 密码哈希
- status: "pending" / "approved" / "rejected"
```

**审批流程**：
1. 用户提交变更申请 → 创建 `UserChangeRequest(pending)`
2. 信号触发为管理员创建系统消息
3. 管理员在站内消息中审批
4. 审批通过后更新 User 表并标记消息已读

#### SystemMessage（系统消息）
```python
- recipient: 接收用户（管理员）
- title / content: 消息内容
- is_read: 已读标记
- related_request: 关联的变更申请
```

### 2.2 核心接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/user/register/` | POST | 用户注册（开放接口） |
| `/api/user/login/` | POST | 用户登录，返回 Token |
| `/api/user/me/` | GET | 获取当前用户信息 |
| `/api/user/me/profile/` | GET/PATCH | 用户中心资料管理 |
| `/api/user/change-password/` | POST | 修改密码 |
| `/api/user/me/sensitive-change/` | POST | 提交敏感信息变更申请 |
| `/api/user/admin/change-requests/` | GET | 管理员审批申请列表 |
| `/api/change-requests/<id>/approve/` | POST | 通过申请 |
| `/api/change-requests/<id>/reject/` | POST | 拒绝申请 |

### 2.3 验证码机制

**实现** (`CaptchaAPIView`)：
1. 生成 4 位随机字符（大写字母+数字）
2. 生成唯一 UUID 标识
3. 存入缓存（Redis/LocMem）5 分钟超时
4. 生成图片（科技蓝配色、白底）
5. 返回 Base64 编码图片 + UUID

**降级策略**：
- Redis 不可用时自动使用内存缓存
- `captcha` 包未安装时返回友好错误提示

```python
- project: 所属项目
```

```python
- test_type: "functional" / "api" / "performance" / "security" / "ui-automation"
```

#### TestCase（测试用例基类）
```python
- module: 所属模块

- case_number: 业务编号（按 test_type 独立递增）
```

#### 性能测试用例扩展 (PerfTestCase)
```python
- concurrency: 并发数
- duration_seconds: 持续时间（秒）
- target_rps: 目标 RPS
```

#### TestCaseStep（用例步骤）
```python
- testcase: 所属用例
- step_number: 步骤编号
```

### 4.3 批量操作逻辑

**批量删除 (`batch_delete`)**：
```python
# 使用 update() 批量更新，避免逐条 delete()
qs.update(is_deleted=True, deleted_at=timezone.now())
```

**批量执行 (`batch_execute`)**：
```python
# 使用 F 表达式原子递增
qs.update(exec_count=F("exec_count") + 1)
```

**物理删除重排 (`hard_delete`)**：
```python
# 删除后批量更新后续编号
TestCase.objects.filter(
    test_type=test_type,
    case_number__gt=seq,
    is_deleted=False,
).update(case_number=F("case_number") - 1)
```

### 5.2 性能任务状态机

```
pending ──run()──> running ──完成──> completed
  │                  │
  │                  └──失败──> failed
  └──直接删除──> (删除)
```

**禁止操作**：
- `running` 状态不能再次触发 `run()`
- `completed` 状态不能直接重跑（建议复制后重跑）

### 5.4 DashboardSummaryAPIView 逻辑

**统计卡片**：
- `totalCases`: 测试用例总数（对比昨日增量）
- `todayExecuted`: 今日执行用例数（今日生成的报告数）
- `unresolvedDefects`: 未解决缺陷数（状态≠4）
- `passRate`: 用例通过率（已完成计划的平均值）

**图表数据**：
- 折线图：近 7 天/30 天执行数与缺陷新增数
- 饼图：按严重程度映射的缺陷分布（1→安全, 2→合规, 3→性能, 4→功能）
- 柱状图：近 7 天缺陷新增数

**活动流**：
- 进行中测试计划（最多 2 条）
- 新缺陷（状态=1，最多 2 条）
- 运行中性能任务（最多 1 条）
- 已完成测试计划（补齐至 6 条）

## 7. AI 助手模块 (`assistant/`)

### 7.1 核心功能

#### 7.1.1 模型连接测试
- `/api/ai/test-connection/` - 使用 OpenAI SDK 测试连接
- `/api/ai/verify-connection/` - 专用连通性校验（固定智谱 v4）
- `/api/assistant/llm/test-connection/` - 原生 HTTP 请求测试

#### 7.1.2 用例生成
- `/api/ai/generate-cases/` - 同步生成用例
- `/api/ai/generate-cases-stream/` - 流式生成用例（SSE）

**生成流程**：
1. 解析请求参数（requirement, api_spec, test_type, module_id）
2. 查询模块内已有用例标题并注入 Prompt（生成前规避重复）
3. 构建系统提示词（含语言强制、格式约束）
4. RAG 检索（可选）查询相似用例
5. 调用 LLM 进行批量生成（单次不少于 5 条）
6. JSON Mode 优先解析（失败回退容错解析）
7. 先做字符串去重，再做向量语义去重
8. 返回清洗后的生成用例列表

**语言强制策略**：
```
【语言强制 · 必须遵守】
- 所有字符串值必须使用简体中文
- JSON 键名必须为 camelCase
- 严禁输出 Markdown 代码围栏
- 仅输出纯 JSON 数组
```

#### 7.1.3 RAG 向量检索
**实现** (`rag_chroma.py`)：
- ChromaDB 本地持久化存储
- 智谱 embedding-3 向量嵌入
- 按 module_id 过滤检索范围

**检索文本构建**：
```python
build_testcase_embed_text(case):
  - 用例名称
  - 等级
  - API 方法+URL（API 类型）
  - 步骤描述 + 预期结果
```

### 7.2 流式生成实现

**SSE 响应格式**：
```json
data: {"type": "connected", "message": "已连接，正在请求模型…", "model": "glm-4.7-flash"}

data: {"type": "meta", "message": "模型请求已发出，正在流式接收内容…"}

data: {"type": "delta", "text": "生成的内容片段"}

data: {"type": "ping", "ts": 1712650000}

data: {"type": "done", "success": true, "message": "已生成 5 条用例", "cases": [...]}
```

**关键特性**：
- 首包立即下发降低感知延迟
- 每 12 秒发送 ping 保持连接
- 未使用 GZipMiddleware（避免 SSE 缓冲问题）

### 7.3 用例去重

**实现** (`generated_case_dedup.py`)：
```python
deduplicate_generated_cases(cases):
  - 基于 caseName + steps 的相似度
  - 保留第一条，删除重复项
```

**语义去重（新增）** (`assistant/services/semantic_dedup.py`)：
```python
semantic_deduplicate_cases(generated_cases, existing_cases):
  - 文本向量化：标题 + 步骤
  - 相似度：Cosine Similarity
  - 阈值：>= 0.85 判定重复并剔除（可配置）
```

## 8. 服务器日志模块 (`server_logs/`)

面向运维与测试的**远程日志查看**：加密保存 SSH 凭据、浏览器端 **WebSocket** 订阅远端 `tail` 输出、划选日志调用 **AI 诊断**（复用平台 `AIModelConfig`）、可选 **Loki** 历史检索，以及**组织级共享**与**操作审计**。详细设计见 **`开发文档/22-服务器日志监控模块开发文档.md`**。

### 8.1 核心模型

#### RemoteLogServer（远程 SSH 主机）
```python
- name / host / port / username
- password_enc / private_key_enc: Fernet 密文
- server_type: "linux" | "windows"
- default_log_path: 默认日志路径（与 WebSocket path 均做安全校验）
- organization: 可选 FK → Organization；绑定后组织创建人与 members 可共用该主机
- 继承 BaseModel：creator / 软删等
```

#### ServerLogAuditEvent（审计事件）
```python
- action: ws_connect / ws_disconnect / ws_stop / analyze / search / host_create / host_update / host_delete
- user / remote_log_server / organization / meta(JSON) / client_ip / created_at
```

#### 与用户模块的协同
- **Organization.members**（M2M）：在「系统管理 → 组织管理」中维护成员后，方可通过组织绑定共享日志主机。

### 8.2 核心接口与通道

| 入口 | 方法 | 说明 |
|------|------|------|
| `/api/server-logs/hosts/` | GET/POST | 主机列表与创建 |
| `/api/server-logs/hosts/<id>/` | GET/PATCH/DELETE | 详情、更新、软删 |
| `/api/server-logs/analyze/` | POST | AI 诊断（限流 `server_logs_analyze`） |
| `/api/server-logs/search/` | GET | Loki `query_range` 代理（未配置则返回未启用） |
| `/api/server-logs/organization-choices/` | GET | 当前用户可绑定的组织（普通成员也可用，区别于 `/api/user/orgs/` 仅系统管理员） |
| `/api/server-logs/audit-events/` | GET | 审计分页；管理员看全部，其余用户仅本人 |
| WebSocket `/ws/server-logs/<id>/` | - | `?token=&path=`；需 **ASGI（如 daphne）**；与 k6 共用 Token 中间件 |

### 8.3 关键逻辑摘要

- **访问范围**：`server_logs/access.py` 统一「创建人 / 绑定组织成员与组织创建人 / 平台管理员」；REST 与 WebSocket **同源规则**，避免列表可见但连接被拒。
- **实时推送**：独立线程执行 Paramiko，`Queue` 异步泵送；队列满时对**日志行**丢弃以免阻塞 SSH 读线程；读超时转 **heartbeat**。
- **前端入口**：主布局菜单「服务器日志」→ 路由 `/server-logs`（`LogTerminal.vue` 等）。

### 9.1 BaseModel（基础模型）

```python
class BaseModel(models.Model):
    creator = ForeignKey(User, null=True, related_name="%(class)s_created")
    updater = ForeignKey(User, null=True, related_name="%(class)s_updated")
    create_time = DateTimeField(auto_now_add=True)
    update_time = DateTimeField(auto_now=True)
    is_deleted = BooleanField(default=False)
    
    class Meta:
        abstract = True
```

**设计特点**：
- 所有业务模型继承此类
- `%(class)s` 动态生成反向关系名
- 软删除统一处理

## 10. 关键业务流程总结

### 10.1 测试用例完整生命周期

```
创建用例
  ↓
填写基本信息（名称、类型、等级）
  ↓
分配编号（按 test_type 独立递增）
  ↓
添加执行步骤（可选）
  ↓
提交评审
  ↓
评审通过 → 归档
  ↓
执行用例
  ↓
记录执行日志（exec_count +1）
  ↓
生成测试报告
```

### 10.2 AI 用例生成流程

```
用户输入需求描述
  ↓
解析参数（test_type, module_id, api_spec）
  ↓
读取模块已有用例标题（前置规避）
  ↓
RAG 检索（可选）查询相似用例
  ↓
构建系统提示词（含语言强制）
  ↓
调用 LLM 批量生成 JSON（>=5 条）
  ↓
解析 JSON 数组
  ↓
字符串去重 + 语义去重
  ↓
返回生成的用例列表
  ↓
用户编辑并保存
```

### 10.3 性能任务执行流程

```
创建性能任务
  ↓
设置任务参数（名称、并发、持续时间）
  ↓
状态: pending
  ↓
点击运行按钮
  ↓
状态: running
  ↓
执行压测脚本（JMeter/Locust）
  ↓
状态: completed / failed
  ↓
查看测试报告
```

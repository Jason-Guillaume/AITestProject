# 31-AI 用例导入全链路：执行优先级与实现状态

## 0. 文档目的

- 规定 **AI 生成 → 预览 → 预检 → 导入 → 导入后校验** 相关能力的 **开发与验收执行顺序**（按依赖关系排序）。
- 汇总各分册文档与 **仓库内真实代码入口**，便于 PM/开发对齐「是否已落地」。
- 本文随功能迭代更新；分册细节仍以 `26`～`30` 为准，本文负责 **索引与状态**。

## 5. 最小验收路径（建议按优先级走一遍）

1. **P0**：同步或流式生成用例 → 响应/SSE `done` 含 `run_id` → 勾选预览行 → `ai-import` 一次成功，`imported[].case_id` 非空，库中 `TestCase.ai_run_id` 可关联（若传 `run_id`）。
2. **P1**：导入 payload 含 `steps_list` → 库中 `TestCaseStep` 条数与列表一致；`module_name` 近重复时复用已有模块（`module_resolution.mode=fuzzy`）。
3. **P2**：预览表故意留空模块 / 缺 API 字段 → 行标红；开启阻断则无法导入；`strict: true` 时后端对问题条进入 `failed[]`。
4. **P3**：API 类型生成结束 → 自动预检回填 → 未替换 `${var}` 能在预览表体现并参与阻断逻辑。
5. **P4**：API 导入成功后弹出「导入后预检报告」→ 列表展示 method/url 与 `unresolved_vars`。
6. **P5**：故意保留 `${token}` 未替换、`strict=true` 且传 `precheck_overrides` → 直连 `ai-import` 仍应在 `failed[]` 中失败（与预检结论一致）。

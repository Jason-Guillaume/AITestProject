# 34-AI能力路线图（闭环修复 / 编排 / 安全 / 知识库 / 治理 / 自动化操作）

## 0. 文档目的

本文件将“我希望 AI 有更多功能”的需求，拆为 **可排期、可验收、可审计** 的能力集合，并与现有实现对齐：

- 现有 AI 能力开发规范：见 `开发文档/09-AI开发指南.md`
- 现有 AI 治理（用量/配额/审计）：见 `开发文档/25-AI治理（用量配额与审计）开发文档.md`
- 现有系统审计（AuditEvent）与导出：见 `开发文档/07-安全与权限基线.md`、`开发文档/04-API接口与数据交互.md`
- 已落地的“失败执行→修订建议”（MVP-A）与“发布简报”（MVP-B）：见 `开发文档/33-后续功能规划-MVP与安全加固.md`

本路线图只定义 **增量能力** 的“做什么 / 怎么做 / 怎么验收 / 怎么审计”，不要求一次性全部实现；请按 §2 分期推进。

---

## 1. 总体原则（企业内网 + 可追溯）

### 1.1 默认可追溯

- **所有敏感非 CRUD 动作**必须写入 `common.models.AuditEvent`（`action=execute/export/create/update` 等），并补足 `extra`（如 run_id / 证据来源 / 导出条件）。
- **AI 的每次调用**必须写入 `assistant.models.AiUsageEvent`（已存在），并对涉及业务资产的动作追加 `meta` 引用（case_id / scenario_id / execution_log_id / defect_id 等）。

### 1.2 默认可审核（Human-in-the-loop）

- AI 输出必须先进入“草稿/建议/变更集”，**默认不直接覆盖生产资产**。
- 所有“一键应用/批量应用”必须二次确认，并留下审计事件（谁在什么条件下应用了 AI 变更）。

### 1.3 默认可回滚

- 自动修复/批量应用应生成 **可回滚的变更单**（见 §3.1 的 `AiPatch` 设计），确保能追溯每次变更差异。

## 2. 推荐分期（MVP → 增强 → 企业级）

> 目标：每期都能“上线带价值”，同时不破坏现有链路（AI 导入、场景执行、审计治理）。

### 2.1 MVP（1～2 周）

- **MVP-1 闭环修复增强**：对“失败执行→修订建议”补齐“差异补丁 / 回滚 / 批量应用门禁”（§3.1）
- **MVP-2 场景编排智能化**：基于 OpenAPI/抓包记录生成最小可执行场景（§3.2）
- **MVP-3 安全测试用例生成（规则优先）**：生成安全用例 + 规则库命中解释（§3.3）
- **MVP-4 企业知识库可追溯问答**：回答必须带引用（§3.4）
- **MVP-5 成本治理增强**：组织/项目维度配额与告警接口（§3.5）
- **MVP-6 自然语言驱动操作（最小可用）**：限定动作集的“AI 命令”执行器（§3.6）

### 2.2 增强（2～6 周）

- 修复建议支持“**根因链路**”：鉴权/变量提取/断言/数据集/环境差异
- 场景编排支持“**参数化 + 数据集**”与失败策略（continue/stop/rollback）
- 安全测试支持“**运行时探测**”（带限流、白名单域名、可回滚）
- 知识库支持“文档分段索引 + 权限过滤 + 脱敏”
- 治理支持“**按组织计费/成本中心**”与“高危操作审批”
- NL 操作支持“多步计划 + 可执行工具调用 + 逐步审计”

### 2.3 企业级（6 周+）

- 变更单（AiPatch）接入审批流：重要资产变更需审批
- 多模型策略：不同场景自动选模型（成本/质量/延迟）
- 安全扫描集成（SAST/DAST）与自动缺陷流转

## 3. 能力块设计（接口 / 数据模型 / 审计点 / 验收）

### 3.1 闭环修复（失败执行 → 补丁 → 一键应用 → 回滚）

#### 3.1.1 目标

把现有 `POST /api/ai/suggest-case-fix/`（已落地）升级为“可落地变更集”，并支持回滚。

#### 3.1.2 建议新增数据模型（最小）

> 放置建议：`assistant/models.py` 或单独 app（若后续审批/版本化很重）。

- `AiPatch`
  - `id`
  - `target_type`：`testcase.TestCase` / `execution.ApiScenario` 等
  - `target_id`
  - `source_execution_log_id`（可空）
  - `status`：`draft/applied/rolled_back/cancelled`
  - `summary`：AI 给出的摘要
  - `changes`：JSON（patch 列表：新增/删除/替换步骤、断言变更、鉴权配置变更等）
  - `risk_level`：`low/medium/high`
  - `created_by` / `created_at`

#### 3.1.3 建议接口

- `POST /api/ai/patches/from-execution/`
  - 入参：`execution_log_id` + 可选 `hint`
  - 出参：`patch_id` + `changes` + `risk_level`
- `POST /api/ai/patches/<id>/apply/`
  - 入参：`confirm=true`、可选 `scope=steps/assertions/auth`
  - 行为：把 patch 应用到目标对象（写库）
- `POST /api/ai/patches/<id>/rollback/`
  - 行为：按 `changes` 逆操作回滚
- `GET /api/ai/patches/?target_type=&target_id=`（列表）

#### 3.1.4 审计点（必须）

- apply / rollback：写 `AuditEvent(action=update)`，`extra={ patch_id, target, scope }`
- patch 生成：写 `AiUsageEvent(action=suggest_case_fix)` 的 `meta.patch_id/target_id`

#### 3.1.5 验收清单

- [ ] patch 生成不落库修改（只生成草稿）
- [ ] patch 应用后，目标用例/场景发生预期变化
- [ ] patch 可回滚，回滚后目标对象与 apply 前一致（关键字段对比）
- [ ] apply/rollback 均有 `AuditEvent`

---

### 3.2 场景编排智能化（从 OpenAPI/抓包 → 场景 → 变量提取/注入 → 参数化）

#### 3.2.1 目标

把“可执行的 API 用例”升级为“可复用的业务链路场景”：登录→查询→创建→校验等，并能在企业内网环境稳定执行。

#### 3.2.2 复用现有能力

- 现有场景执行：`execution/services/scenario_runner.py`（已落地）
- 变量注入/鉴权：`testcase/services/auth_runtime.py`（已落地）
- 断言 DSL：`testcase/services/assertions.py`（已落地）

#### 3.2.3 建议新增能力

- **场景生成器**：从 OpenAPI（或 HAR/curl）解析 endpoint，推断“必要前置”（如登录）并生成步骤序列
- **变量策略模板**：
  - 自动识别 token/session 字段 → 写入 `extract`（变量提取）
  - 后续步骤自动引用 `{{token}}` / `{{user_id}}` 等
- **数据集（Dataset）**：为同一场景提供多组参数（正常/边界/逆向）

#### 3.2.4 建议接口

- `POST /api/execution/api-scenarios/generate/`
  - 入参：`project_id`、`env_id`、`openapi_spec` 或 `har`/`curl_list`
  - 出参：`scenario_draft`（不落库）或 `scenario_id`（落库需要 confirm）
- `POST /api/execution/api-scenarios/<id>/datasets/`（可选）

#### 3.2.5 审计点

- 生成场景：`AuditEvent(action=execute, object=ApiScenario)`（extra 记录来源：openapi/har）
- 落库创建：走 `BaseModelViewSet` 的 create 审计 + extra 扩展

#### 3.2.6 验收清单

- [ ] 给定 OpenAPI（含鉴权说明）可生成可执行场景草稿
- [ ] 生成结果能直接用 runner 运行，并产出 run/step_run
- [ ] 变量提取/注入在至少 2 个步骤间生效

---

### 3.3 安全测试 AI（生成用例 + 规则解释 + 自动缺陷）

#### 3.3.1 目标

把“安全测试”从纯生成扩展为“可解释、可复现、可流转”的结果：命中规则 → 给出证据 → 生成缺陷。

#### 3.3.2 设计取向（先规则、后模型）

- MVP 优先：**规则库（OWASP API Top 10 / 常见越权/注入/敏感泄露）** + 少量 AI 扩写
- 运行时探测（增强期）：需要白名单域名、限流、敏感字段脱敏

#### 3.3.3 建议接口

- `POST /api/ai/security/generate-cases/`
  - 入参：`openapi_spec` + `scope`（auth/idor/injection/xxe…）
  - 出参：`cases[]`（结构化 + 风险等级 + 命中规则）
- `POST /api/ai/security/analyze-execution/`
  - 入参：`execution_log_id`
  - 出参：`findings[]`（rule_id、evidence、suggest_fix）
- `POST /api/defect/defects/from-security-finding/`（可选）

#### 3.3.4 审计点

- 生成/分析：写 `AiUsageEvent(action=security_generate/security_analyze)`
- 自动建缺陷：写 `AuditEvent(action=create)`（extra 记录 finding_id 与证据引用）

#### 3.3.5 验收清单

- [ ] 输出包含 rule_id 与可读解释（why / how to reproduce）
- [ ] 能从 execution_log 推导出最小证据（请求/响应片段脱敏后）

---

### 3.4 企业知识库可追溯问答（带引用、带权限、带脱敏）

#### 3.4.1 目标

让 AI 问答变成“企业内部的可追溯知识查询”，输出必须引用来源，避免幻觉。

#### 3.4.2 复用现有能力

- 知识库与 RAG：见 `开发文档/18-RAG测试知识库开发文档.md`

#### 3.4.3 建议新增：引用协议（强约束）

统一在 AI 响应中返回：

- `answer_markdown`
- `citations[]`：`{ type: "doc"|"defect"|"case"|"audit", id, title, url }`

并在前端以“可点击引用”展示（引用必须能跳转到系统对象详情页）。

#### 3.4.4 建议接口

- `POST /api/ai/knowledge/ask/`
  - 入参：`question` + 可选 `project_id/module_id`
  - 出参：`answer_markdown` + `citations[]`

#### 3.4.5 权限与脱敏

- 默认按项目成员过滤可引用对象；系统管理员可全局查询
- 引用内容与检索内容在入模前做脱敏（token/手机号/身份证/邮箱/密钥）

#### 3.4.6 验收清单

- [ ] 每次回答至少 1 条 citation（或明确“无可引用来源”并返回空 citations）
- [ ] citations 点击可跳转到系统对象

---

### 3.5 成本治理（组织/项目维度配额、告警、策略）

#### 3.5.1 目标

在现有“按用户配额/并发”的基础上，增加企业常见维度：

- 组织级：某组织每日上限
- 项目级：某项目每日上限
- 策略级：高成本 action（如大上下文 RAG / 安全分析）更严格

#### 3.5.2 建议新增模型

- `AiQuotaPolicy`
  - `scope_type`：`org/project/user`
  - `scope_id`
  - `daily_requests`、`max_concurrency`
  - `allowed_actions`（可选白名单）
  - `created_by/updated_by`

#### 3.5.3 建议接口（系统管理员）

- `GET/PUT /api/sys/ai-quota/`（全局/按 scope）
- `GET /api/sys/ai-usage/alerts/`（可选：告警聚合）

#### 3.5.4 审计点

- 配额策略变更：`AuditEvent(action=update, object=AiQuotaPolicy)`

#### 3.5.5 验收清单

- [ ] org/project 配额触发时返回 429，并写 AiUsageEvent（success=false、error_code=quota）
- [ ] 管理员可查询策略与近 7 天超限次数

### 3.6 自然语言驱动操作（AI 命令执行器）

#### 3.6.1 目标

让用户用一句话驱动平台完成多步动作，但必须：

- 限定可执行动作集合（避免“任意执行”）
- 每一步可审计、可回滚、可中断

#### 3.6.2 建议设计：Tool Registry（白名单）

新增 `assistant/tools/`（或 `assistant/services/tools_*.py`）：

- 每个 tool 明确：
  - `name`
  - `input_schema`
  - `permission_check(request.user, input)`
  - `execute(input) -> output`

MVP tool 示例（建议从只读/低风险开始）：

- `list_release_risk_brief(release_id, days)`
- `export_never_executed_cases_csv(release_id, days)`
- `run_api_scenario(scenario_id)`
- `create_defect_from_execution(execution_log_id)`

#### 3.6.3 建议接口

- `POST /api/ai/ops/execute/`
  - 入参：`command`（自然语言）+ 可选 `context`（project/env）
  - 出参：`plan[]`（解释将执行哪些 tool）+ `results[]`

#### 3.6.4 审计点

- 每个 tool 的执行都必须写 `AuditEvent(action=execute)`（object 指向被操作对象）
- AI 总调用写 `AiUsageEvent(action=ops_execute)`，`meta.tools=[...]`

#### 3.6.5 验收清单

- [ ] 非管理员无法执行管理员 tool（403）
- [ ] 命令会先返回 plan，再执行（或一次性返回但包含 plan 与 results）
- [ ] 每个子动作均可在审计中心检索到

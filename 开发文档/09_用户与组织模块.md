# 用户与组织模块（user）

## 功能概述
负责系统用户的身份认证、权限控制、组织（Company）管理以及系统消息、验证码、AI 模型配置等配套功能。提供用户注册/登录、密码修改、个人信息管理、组织成员管理、系统消息推送、AI 使用额度配置等完整的后台管理能力。

## 使用技术
- **Django ORM**：`User` 继承自 `AbstractUser` 并混入 `BaseModel`，拥有审计字段与软删。`Organization`、`SystemMessage`、`AIModelConfig`、`AiQuotaPolicy` 等模型实现组织、系统消息及 AI 配置。
- **Django REST Framework (DRF)**：`UserViewSet`、`OrganizationViewSet`、`SystemMessageSettingViewSet` 等 ViewSet 提供统一的 CRUD 接口；`APIViews`（如 `UserRegisterAPIView`, `UserLoginAPIView`）处理登录、注册等自定义流程。
- **权限系统**：`permissions.py` 中的 `IsSystemAdmin`, `IsApprovalAdmin` 等自定义 Permission 类实现细粒度访问控制。
- **验证码**：`captcha_image.py` 使用 `ImageCaptcha` 生成图形验证码，防止暴力登录。
- **审计日志**：`AuditEvent`（在 `common/models.py`）记录用户关键操作，配合 `AuditEventListAPIView` 提供查询。
- **Celery（可选）**：部分异步操作如系统消息批量发送、AI 使用上报可通过 Celery 完成。

    content = models.TextField()

    model_type = models.CharField(max_length=50)
    api_key = models.CharField(max_length=255)
    base_url = models.URLField(blank=True, null=True)

### 序列化器 (`user/serialize.py`)
```python
class UserSerializer(BaseModelSerializers):
    class Meta:
        model = User
        fields = ('id', 'username', 'email', 'is_active', 'date_joined')

class UserRegisterSerializer(serializers.Serializer):
    username = serializers.CharField()
    password = serializers.CharField(write_only=True)
    email = serializers.EmailField()
    def create(self, validated_data):
        user = User.objects.create_user(**validated_data)
        return user

class OrganizationSerializer(BaseModelSerializers):
    class Meta:
        model = Organization
        fields = '__all__'
```

### 视图（核心 API） (`user/views.py`)
```python
class UserViewSet(BaseModelViewSet):
    queryset = User.objects.all()
    serializer_class = UserSerializer
    permission_classes = [IsSystemAdmin]

class UserRegisterAPIView(APIView):
    def post(self, request):
        serializer = UserRegisterSerializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        user = serializer.save()
        return ApiResponse.created(UserSerializer(user).data)

class UserLoginAPIView(APIView):
    def post(self, request):
        serializer = UserLoginSerializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        token = serializer.validated_data['token']
        response = ApiResponse.success({'token': token})
        # 将 token 写入 HttpOnly Cookie，前端通过 header 读取
        response.set_cookie(key='access_token', value=token, httponly=True)
        return response

class OrganizationViewSet(BaseModelViewSet):
    queryset = Organization.objects.all()
    serializer_class = OrganizationSerializer
```

## 为什么使用这些技术
- **统一基类** 让用户、组织等模型拥有统一的审计字段，便于日志审计与软删。
- **DRF + 自定义 APIView** 兼顾标准化 CRUD 与特殊业务（登录、验证码、租户配额）。
- **Permission** 确保只有系统管理员或组织管理员可以进行敏感操作。
- **验证码** 增强登录安全性，防止机器暴力破解。
- **Celery** 可将系统消息、AI 使用上报等耗时操作异步化，提高响应速度。

## 常见错误与排查
1. **登录失败**：检查 `UserLoginSerializer` 中的密码校验是否使用 `check_password`，并确认 `AUTHENTICATION_BACKENDS` 已包含默认后端。
2. **验证码失效**：验证码字符保存在 session 中，若跨域请求未携带相同 session，验证会失败。可改用 Redis 存储或在前端同域请求。
3. **组织成员未同步**：`on_org_members_changed` 信号在 `user/signals.py` 中处理成员变更，确保 `signals.py` 已在 `UserConfig.ready()` 中导入。
4. **权限 403**：`IsSystemAdmin` 只允许 `request.user.is_superuser`，普通用户需要管理员角色才能访问对应接口。
5. **AI 模型配置错漏**：`AIModelConfig` 保存的 API Key 必须在 `settings.py` 中对应的环境变量里设置，否则调用 OpenAI/Zhipu 时会报 401。

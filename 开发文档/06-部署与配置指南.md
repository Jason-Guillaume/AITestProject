# 06-部署与配置指南

## 0. 需求来源与开发动因

- 业务价值摘要：标准化部署流程，降低环境差异导致的发布故障。
- 业务背景：系统同时覆盖本地开发、测试环境与生产部署，需要统一交付标准。
- 现状痛点：环境差异和配置漂移易造成“本地可用、线上失败”，排查成本高且恢复慢。
- 建设目标：提供标准化部署手册与配置基线，统一发布流程。
- 预期收益：降低发布风险，提升运维效率与可观测性。

## 1. 部署目标与适用范围

本文档用于指导 SmartTest 在不同环境（本地开发、测试环境、生产环境）下的配置与部署。内容覆盖：

- Django 后端配置项与环境变量治理
- 前端构建与反向代理策略
- 数据库/缓存/媒体文件配置
- AI 模型连接配置与安全注意事项
- 上线前检查清单与常见问题排查

### 2.1 默认端口规划

## 3. 后端配置说明（Django）

核心配置文件：`AITestProduct/settings.py`

### 3.1 基础配置项

| 配置项 | 当前默认值 | 生产建议 |
|------|------|------|
| `DEBUG` | 环境变量 `DJANGO_DEBUG`（默认 `True`） | 生产必须为 `0/False` |
| `ALLOWED_HOSTS` | `[]` | 明确配置域名/IP 白名单 |
| `SECRET_KEY` | 环境变量 `DJANGO_SECRET_KEY`（开发缺省将使用临时值） | 生产必须通过环境变量注入，禁止入库 |
| `TIME_ZONE` | `UTC` | 按团队实际需求设置（如 `Asia/Shanghai`） |

### 3.2 数据库配置

当前项目默认使用 MySQL，配置在 `DATABASES["default"]`。

建议将以下字段全部改为环境变量：

- `NAME`
- `USER`
- `PASSWORD`
- `HOST`
- `PORT`

示例（推荐模式）：

```python
DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.mysql",
        "NAME": os.environ.get("DB_NAME", "ai_test_product"),
        "USER": os.environ.get("DB_USER", "root"),
        "PASSWORD": os.environ.get("DB_PASSWORD", ""),
        "HOST": os.environ.get("DB_HOST", "127.0.0.1"),
        "PORT": os.environ.get("DB_PORT", "3306"),
    }
}
```

### 3.2.1 生产安全启动校验（新增）

为避免“误把开发配置带到生产”，后端在 **`DJANGO_DEBUG=0`** 时会强制校验：

- `DJANGO_SECRET_KEY` 必须存在，否则启动直接失败
- `DB_PASSWORD` 必须存在，否则启动直接失败
- 若设置了 `ELASTICSEARCH_USER`，则 `ELASTICSEARCH_PASSWORD` 必须存在，否则启动直接失败

该策略用于阻断高风险的“明文默认值上线”。

### 3.3 缓存与 Redis 降级策略

项目内置了“Redis 可用性探测 + 自动降级 LocMem”逻辑：

1. `USE_REDIS_CACHE=1` 且 Redis 可连通时，启用 `django_redis`
2. 否则自动降级为 `LocMemCache`

相关环境变量：

| 变量名 | 默认值 | 说明 |
|------|------|------|
| `USE_REDIS_CACHE` | `1` | 是否尝试 Redis |
| `REDIS_HOST` | `127.0.0.1` | Redis 地址 |
| `REDIS_PORT` | `6379` | Redis 端口 |

### 3.4 鉴权与权限基线

DRF 全局策略：

- 认证：`TokenAuthentication`
- 权限：`IsAuthenticated`

含义：除显式开放接口（登录、注册、验证码）外，接口默认需登录。

### 3.5 媒体文件配置

| 配置项 | 当前值 | 说明 |
|------|------|------|
| `MEDIA_URL` | `/media/` | 访问前缀 |
| `MEDIA_ROOT` | `BASE_DIR / "media"` | 文件落盘路径 |

开发环境下，`urls.py` 在 `DEBUG=True` 时由 Django 直接托管媒体文件；生产建议改由 Nginx 托管。

### 3.6 登录兼容开关（历史密码迁移）

项目支持以下兼容开关（默认关闭）：

- `LEGACY_LOGIN_MD5_HEX`
- `LEGACY_LOGIN_SHA256_HEX`
- `LEGACY_LOGIN_BOTH_SIDES_MD5_HEX`
- `LEGACY_LOGIN_PLAINTEXT`

建议仅在历史数据迁移窗口临时开启，迁移完成后全部关闭。

### 3.7 邮件配置

用户中心敏感信息审批通知使用 Django 邮件能力，关键项：

| 变量名 | 说明 |
|------|------|
| `EMAIL_BACKEND` | 邮件后端（开发默认 console） |
| `DEFAULT_FROM_EMAIL` | 发件地址 |
| `EMAIL_SUBJECT_PREFIX` | 邮件标题前缀 |
| `USER_CENTER_ADMIN_EMAIL` | 审批通知收件人（可逗号分隔） |

## 5. 环境变量清单（建议统一治理）

虽然当前项目尚未提供 `.env` 模板文件，但建议按“分环境 + 最小暴露”原则建立：

- `.env.dev`
- `.env.test`
- `.env.prod`

建议至少包含以下变量：

| 分类 | 变量名 | 说明 |
|------|------|------|
| Django 安全 | `DJANGO_SECRET_KEY` | 替代代码内密钥 |
| Django 运行 | `DJANGO_DEBUG` | 开发 `1`，生产 `0` |
| Django 主机 | `DJANGO_ALLOWED_HOSTS` | 逗号分隔白名单 |
| 数据库 | `DB_NAME/DB_USER/DB_PASSWORD/DB_HOST/DB_PORT` | MySQL 连接信息 |
| 缓存 | `USE_REDIS_CACHE/REDIS_HOST/REDIS_PORT` | Redis 策略 |
| 邮件 | `EMAIL_BACKEND/DEFAULT_FROM_EMAIL/USER_CENTER_ADMIN_EMAIL` | 审批通知 |
| 密码兼容 | `LEGACY_LOGIN_*` | 历史迁移临时开关 |
| AI 生成 | `AI_CASE_BATCH_MIN_COUNT` | AI 单次批量生成最小条数（默认 5） |
| AI 去重 | `AI_CASE_SEMANTIC_DUP_THRESHOLD` | 语义去重阈值（默认 0.85） |

## 6. 分环境配置差异建议

### 6.1 本地开发（dev）

推荐特征：

- `DEBUG=True`
- `ALLOWED_HOSTS=[]` 或本机地址
- 可使用 LocMem 缓存降级
- 前端通过 Vite 代理访问后端
- 邮件后端使用 `console.EmailBackend`

### 6.2 测试环境（test/staging）

推荐特征：

- `DEBUG=False`
- 独立 MySQL/Redis 实例
- 打开 CI 自动化测试（pytest smoke/regression）
- 配置可用但隔离的 AI API Key
- 校验 AI 批量生成与语义去重配置值符合预期
- 保留详细日志与监控告警

### 6.3 生产环境（prod）

推荐特征：

- `DEBUG=False`
- 强密码与最小权限数据库账号
- HTTPS 与反向代理
- 媒体文件由 Nginx 或对象存储托管
- 关闭全部 `LEGACY_LOGIN_*` 兼容开关
- AI 配置通过后台 `/api/sys/ai-config/` 由系统管理员维护

## 7. 部署步骤（推荐流程）

### 7.1 后端部署

```bash
# 1. 安装依赖
pip install -r requirements.txt

# 2. 执行迁移
python manage.py migrate --noinput

# 3. （可选）初始化演示性能任务
python manage.py seed_perf_tasks

# 4. 启动服务（示例）
python manage.py runserver 0.0.0.0:8000
```

生产建议改为 Gunicorn/Uvicorn + 进程守护（systemd/supervisor）。

### 7.2 前端部署

```bash
cd frontend
npm install
npm run build
```

将 `frontend/dist/` 发布到静态资源服务器，并配置：

- `/` -> 前端静态页面
- `/api` -> 后端网关地址
- `/media` -> 后端或媒体服务地址

### 7.3 Nginx 关键点（示意）

1. 静态资源开启缓存策略
2. `/api` 与 `/media` 反向代理到 Django
3. 对 AI 流式接口适当放宽超时
4. 开启 HTTPS

## 8. 上线前检查清单

### 8.1 安全检查

- [ ] `DEBUG` 已关闭
- [ ] `SECRET_KEY` 不在仓库明文出现
- [ ] 数据库密码改为环境变量
- [ ] `ALLOWED_HOSTS` 已正确设置
- [ ] 管理员账号密码已重置为强密码

### 8.2 可用性检查

- [ ] `python manage.py migrate` 执行成功
- [ ] 核心接口（登录、项目、用例、执行、缺陷）可访问
- [ ] `/api/ai/test-connection/` 或 `/api/ai/verify-connection/` 可正常返回
- [ ] `/api/ai/generate-cases/` 返回多条用例且格式为合法 JSON 数组语义
- [ ] AI 去重生效：重复意图用例会被过滤
- [ ] 前端页面可正常调用 `/api` 与加载 `/media`

### 8.3 观测与回滚

- [ ] 关键日志已接入（应用日志、Nginx 日志）
- [ ] 审计日志可回溯关键动作（创建/更新/删除/执行/导出）
- [ ] 备份策略已确认（DB + media）
- [ ] 回滚版本包可快速切换

审计最小复测（建议上线前做一次）：
- 以普通用户登录后做一次业务动作（例如触发场景执行 `POST /api/execution/api-scenarios/<id>/run/` 或创建缺陷）；
  然后访问：`GET /api/user/me/audit/events/` 应能看到对应记录。
- 以系统管理员登录后访问：`GET /api/sys/audit/events/?limit=200`，应能查询到全量审计事件，并可按 `action` 过滤（如 `export/execute`）。

## 9. 常见部署问题排查

| 问题 | 现象 | 排查建议 |
|------|------|------|
| `401 Unauthorized` | 登录后接口仍未授权 | 检查前端是否写入 `localStorage.token`，请求头是否带 `Token xxx` |
| 静态页面正常但 API 404 | `/api/*` 不通 | 检查 Nginx 反向代理路径与后端服务地址 |
| 图片无法访问 | `/media/*` 404 | 检查 `MEDIA_ROOT` 挂载与 Nginx `/media` 路由 |
| 验证码接口报错 | `/api/user/captcha/` 500 | 检查 `captcha` 依赖与缓存后端配置 |
| AI 流式中断 | 前端 SSE 提前断开 | 检查代理超时、反向代理缓冲策略、上游模型连通性 |

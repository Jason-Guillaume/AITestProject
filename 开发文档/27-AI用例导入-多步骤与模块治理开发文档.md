# 27-AI用例导入：多步骤与模块命名治理开发文档

## 0. 需求来源与开发动因

- **业务价值摘要**：提升 AI 导入用例的可维护性（多步骤）与资产整洁度（模块树不被污染）。
- **现状痛点**
  - 导入后每条用例默认只有 1 个 step，步骤粒度粗、后续编辑成本高。
  - AI 产出的 `module_name` 易出现近重复/同义模块，导致模块树膨胀。
- **建设目标**
  - 批量导入接口支持 `steps_list[]`，可创建多个 `TestCaseStep`
  - 模块解析增加“相似匹配优先复用”，仅在无匹配时创建新模块

---

## 1. 能力范围（本期）

### 1.1 多步骤导入

批量导入接口 `POST /api/testcase/cases/ai-import/` 新增支持：

- `steps_list`（或 `stepsList`）：数组形式的步骤
- 兼容：若未提供 `steps_list`，仍按旧字段 `precondition/steps/expected_result` 生成单 step

同时，AI 生成侧已升级 schema：

- 批量生成 Prompt 会要求模型尽量输出 `steps_list[]`
- 后端归一化会把 `steps_list` 保留在 `cases[]` 中；若 `steps` 为空，会从 `steps_list` 自动拼出可读的 `steps` 文本用于兼容展示

### 1.2 模块命名治理（相似匹配复用）

当 `module_name` 不是“精确命中”已存在模块时：

- 使用相似度（`SequenceMatcher.ratio()`）在同项目同 `test_type` 的模块中寻找最相近项
- 相似度达到阈值则直接复用该模块，不再新建
- 阈值可由请求体配置：`module_match_threshold`（默认 `0.92`，范围 0.70~0.99）

导入结果中返回：

- `module_resolution`：`{ mode, matched_name, score }`
  - `exact`: 精确命中
  - `fuzzy`: 相似匹配复用
  - `created`: 新建模块
  - `explicit`: 前端显式选择 module_id（导入前可视化纠偏）

同时前端提供“模块匹配可视化”：

- 在 AI 预览表中每行显示一个可选下拉框（模块列表）
- 用户可在导入前明确指定目标模块（传 `item.module_id`），后端将优先使用该 module_id

## 2. 接口说明

### 2.1 `POST /api/testcase/cases/ai-import/`（增强）

新增请求字段：

- `module_match_threshold`：可选，默认 0.92

单条 item 新增/兼容字段：

- `steps_list` / `stepsList`：可选，数组
  - 每个元素建议为对象：`{ step_desc, expected_result }`
  - 兼容：`{ desc }` / `{ action }` / `{ expected }`

响应新增字段：

- `imported[].steps_created`：本条创建的 step 数
- `imported[].module_resolution`：模块解析结果（见上文）

## 4. 验收清单（最小）

- [x] `POST /api/ai/generate-cases/` 或流式生成返回的 `cases[]` 中包含 `steps_list`（当模型输出时）
- [x] item 中传入 `steps_list`，导入后用例创建多个 `TestCaseStep`（step_number 递增）
- [x] `module_name` 与已有模块相似度足够高时，导入不会新建模块，而是复用（`module_resolution.mode = fuzzy`）
- [x] `module_match_threshold` 可调整：提高阈值会减少复用、降低阈值会增加复用

最小复测路径：按 `开发文档/31` §5 步骤 2（P1）走一遍即可。
